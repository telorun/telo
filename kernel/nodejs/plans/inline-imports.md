# Inline Imports

## Problem

Every dependency is declared as its own top-level `Telo.Import` document. Once [export-resource-instances.md](./export-resource-instances.md) lands, even a zero-config dependency leaves boilerplate behind ‚Äî hello-world is two documents where the second is pure wiring:

```
targets:
  - invoke: !ref Console.writeLine
    inputs: { output: "Hello World!" }
---
kind: Telo.Import
metadata: { name: Console }
source: std/console@0.8.0
```

An import is **module wiring**, not a runtime resource: it has no capability, never appears in `resources.<name>` as a node, and is already excluded from the dependency graph ([system-kinds.ts](../../../analyzer/nodejs/src/system-kinds.ts) `DEPENDENCY_GRAPH_SKIP_KINDS`). Yet it is the only piece of wiring expressed as a separate document ‚Äî `variables`, `secrets`, `ports`, `include`, `targets`, `exports` are all inline properties of the module doc. Imports belong in that family.

## Solution

Add an optional `imports:` map to `Telo.Application` / `Telo.Library` ‚Äî additive: authored `Telo.Import` documents keep loading unchanged, both forms may coexist, and inline becomes the idiomatic form once the CLI follow-up (below) can manage it.

**Shape.** A name-keyed map whose key is the PascalCase alias (replacing today's `metadata.name`), each entry carrying the same fields the `Telo.Import` doc carries minus the name: `imports: { Console: { source, variables, secrets, runtime } }`. This matches how `variables` / `secrets` / `ports` are already keyed. An entry value may also be a bare **source string** as shorthand for `{ source: <string> }` ‚Äî `imports: { Console: std/console@0.8.0 }` ‚Äî for the common case of a dependency with no `variables` / `secrets` / `runtime`. The same scalar-or-object union the `targets` field already accepts.

**Desugaring is the load-bearing mechanism ‚Äî an opt-in load option, on for every *resolved* consumer, off for the editor's raw view.** The expansion is a single pure transform that first normalizes any scalar entry to `{ source: <string> }`, then turns each app/lib `imports:` entry into a synthetic `Telo.Import` manifest appended to that file's manifest list. It is gated by a new `LoadOptions` flag (alongside the existing `compile`) honored where a file's parsed manifests are assembled in the [manifest-loader.ts](../../../analyzer/nodejs/src/manifest-loader.ts) `Loader`, so it propagates uniformly through `loadFile` ‚Üí `loadModule` ‚Üí `loadGraph` and is folded into the `fileCache` key next to `compile` (a desugared and a raw load of the same file never collide).

Critically, it must feed **both** kernel consumption paths, not just analysis:

- **Analysis** ‚Äî `loadGraph` ‚Üí `flattenForAnalyzer` ([kernel.ts:303-308](../src/kernel.ts#L303-L308)), the path shared with the analyzer. Discovery (fetching transitive libraries, building the alias‚Üísource edge map, validating targets are Libraries) reads `Telo.Import` manifests *during* the `loadGraph` walk, so the synthetics must already be present when it runs ‚Äî hence expansion at file-assembly time, not in the later inline-normalization pass (which would leave discovery blind and make cross-module kinds and `!ref Alias.x` fail static resolution).
- **Runtime** ‚Äî `loadModule` + `flattenLoadedModule` ([kernel.ts:377-378](../src/kernel.ts#L377-L378)), the entry-only runtime load that does **not** go through `loadGraph`. Its manifests are what the kernel's `registerManifest` loop ([kernel.ts:395-434](../src/kernel.ts#L395-L434)) registers, and registration is what makes the import-controller actually run. An authored `Telo.Import` survives here because it is a physical doc; an inline entry produces no doc, so without the option on *this* load the imports would pass static analysis and then **never execute** ‚Äî the app would boot with no `Console`. This is the gap the earlier revisions missed by scoping the desugar to `loadGraph` alone.

The **editor** enables the option nowhere. Its round-trip surfaces ‚Äî the direct `parseLoadedFile` call ([yaml-document.ts](../../../apps/telo-editor/src/yaml-document.ts)) and its own `flattenLoadedModule` ‚Üí `buildParsedManifest` path ([loader.ts](../../../apps/telo-editor/src/loader.ts) ‚Üí [parse.ts](../../../apps/telo-editor/src/loader/parse.ts)) ‚Äî read the raw `imports:` map, because the editor pairs each manifest to its backing YAML `Document` **by index** to round-trip a file ([ast-ops.ts](../../../apps/telo-editor/src/loader/ast-ops.ts) `findDocForResource` / `resourceDocIndex` / `applyOpsToDocument`) and a synthetic import has no backing node. So `flattenLoadedModule` stays a pure projection ‚Äî the synthetics arrive from the gated load *upstream*, never from inside the shared flatten that both the kernel runtime and the editor call. The boundary is three-way: **desugared** for the kernel's analysis load, the kernel's runtime load, and the analyzer; **raw** for the editor's edit/round-trip. "One shared transform" means one transform, selected by one flag ‚Äî not one fixed insertion point.

Each synthetic import carries the **source position** of its `imports:` map entry (the scalar or object YAML value node), so resolution diagnostics ‚Äî bad `source`, missing required `variables`, importing an Application ‚Äî land on the authoring line rather than a phantom document. Relative `source:` paths resolve unchanged because the synthetic import rides in the resolution manifest list stamped with the declaring file URL by the existing flatten/stamp step. Everything else downstream is reused as-is: alias registration, the multi-pass create/init interleave with defer-and-retry, graph exclusion, teardown cascade, and the export-instances `!ref Alias.x` / `exports.resources` resolution (the alias is identical regardless of declaration site).

**Duplicate-alias diagnostic ‚Äî a new validation step, not a free consequence.** Today the alias edge map ([manifest-loader.ts](../../../analyzer/nodejs/src/manifest-loader.ts) `importEdges`, via `aliases.set(alias, ‚Ä¶)`) is plain last-writer-wins with no check anywhere, so two imports sharing an alias load silently under shadowing. This change adds a real check in discovery ‚Äî where the edge map is built, after both authored docs and synthetic-from-`imports` are in the resolution list ‚Äî emitting a **`DUPLICATE_IMPORT_ALIAS`** diagnostic pinned to the second declaration. It fires for *any* repeated alias in a module's scope: two authored `Telo.Import` docs, an owner + included-partial collision, or one authored + one inline (a YAML map can't repeat its own key). This is a genuine behaviour change ‚Äî manifests that previously loaded under silent shadowing now error ‚Äî accepted because silent alias shadowing is a latent correctness bug and the project is pre-release. Called out as its own step rather than folded into "reuse discovery verbatim."

**Editor reconciliation.** The editor reads both representations (authored docs and the raw `imports:` map) into one unified import list. For **writing**, until the CLI follow-up lands it keeps emitting authored `Telo.Import` documents ‚Äî so every tool-created import stays CLI-manageable (see Follow-up) ‚Äî and edits each existing import in place in whatever form it already lives (upgrading an authored doc rewrites that doc's `source:`; an inline entry's `source:` is rewritten in the map). It never silently migrates one form to the other. When the CLI follow-up ships, the editor's write form and the docs' recommendation flip to inline together.

**The guardrail ‚Äî execution stays put.** [import-controller.ts](../src/controllers/module/import-controller.ts) remains the *sole* import execution unit (child-context creation, alias registration, teardown). The expansion is a thin syntactic step in the module/load layer; [module-controller.ts](../src/controllers/module/module-controller.ts) does **not** absorb import behaviour. Declaration moves; execution does not ‚Äî so the Application gains a section, not a responsibility.

**Schemas.** The Application/Library schemas in [builtins.ts](../../../analyzer/nodejs/src/builtins.ts) gain the `imports` map; the kernel mirror in [manifest-schemas.ts](../src/manifest-schemas.ts) follows. The existing `Telo.Import` schema is untouched.

**Editor.** The rich import UI ‚Äî registry search and version upgrade, today in the sidebar `ImportsSection` ‚Äî moves to the **right-side detail page of the module-root node**, rendered per app/lib kind (alongside the variables/secrets editing added in #141), via [parse.ts](../../../apps/telo-editor/src/loader/parse.ts) and [application-adapter.ts](../../../apps/telo-editor/src/application-adapter.ts). Read/merge/write follows the reconciliation rule above (read both forms; write authored docs until the CLI follow-up, then flip to inline; never silently migrate). When it does write inline it writes the full object form ‚Äî the scalar shorthand is an authoring convenience, not something the editor round-trips. Topology is unaffected ‚Äî imports were never nodes.

**Docs & changeset.** Update the import section of [CLAUDE.md](../../../CLAUDE.md), the kernel module docs, and the getting-started example to show the inline form; a changeset covers kernel, analyzer, and editor.

## Follow-up PR ‚Äî CLI

The CLI `install` / `publish` / `upgrade` commands ([cli/nodejs/src/commands](../../../cli/nodejs/src/commands)) read and rewrite `Telo.Import` documents in place and cannot see or manage the inline `imports:` map. A **separate follow-up PR** teaches them the map ‚Äî discovery, add-on-install, and in-place `source:` upgrade against a map entry ‚Äî and **at the same time** flips the editor's write form and the getting-started docs to inline.

The split is deliberate so the recommended / tool-written path is never the unmanaged one. In this PR the inline form is fully usable by the kernel and analyzer and editable by hand, but the editor keeps **writing** authored `Telo.Import` docs and the docs keep **recommending** them, so every import created by the tooling stays CLI-manageable. Hand-authored inline imports run fine; they are simply not CLI-upgradeable until the follow-up. The follow-up then moves the recommendation and the editor's pen to inline in lockstep with CLI awareness, so the default path and the managed path are always the same one. IDE-support `source:` completions inside the inline map ride the same follow-up.

## Decisions

- **Additive and backwards compatible ‚Äî no migration, no codemod.** Both forms coexist; the analyzer does not reject authored `Telo.Import` docs. Rejected a hard cutover + repo-wide codemod: needless churn across ~160 manifests and a break for external manifests, for a purely ergonomic win.
- **Name-keyed map, alias as key.** Mirrors `variables` / `secrets` / `ports`; the key is the alias, dropping the awkward `metadata.name`-as-alias. Rejected a list with an explicit `name`/`alias` field ‚Äî less consistent and more verbose.
- **Scalar-source shorthand.** A bare string entry desugars to `{ source }`, normalized at the front of the parse-step transform so all downstream stages see only the object form. Mirrors the `targets` scalar-or-object union and the Cargo `dep = "1.0"` idiom. The editor sidesteps the union entirely by always writing the full object form.
- **Desugar as one opt-in load option feeding both kernel loads + the analyzer ‚Äî not the editor's raw view.** The synthetic `Telo.Import` manifests must reach the kernel's analysis load (`loadGraph` ‚Üí `flattenForAnalyzer`) *and* its runtime load (`loadModule` ‚Üí `flattenLoadedModule`, the path that registers and executes them ‚Äî missed by earlier revisions that scoped the desugar to `loadGraph`), plus the analyzer; gated by a `LoadOptions` flag folded into the `fileCache` key. The editor enables it nowhere, so its index-paired round-trip (`parseLoadedFile`, `buildParsedManifest`) reads the raw `imports:` map and never sees a synthetic. Rejected: expanding in `parse-loaded-file.ts` (shared with the editor's raw view ‚Üí breaks index pairing), in the later inline-normalization pass (discovery goes blind ‚Üí false unresolved-reference errors), or in `loadGraph` only (analysis-only ‚Üí imports analyze but never execute); runtime creation by the module-controller (delicate mid-loop ordering); and letting the module-controller own import execution (the god-controller risk this design exists to avoid).
- **Duplicate-alias is a new diagnostic and an accepted behaviour change.** A new check in discovery emits `DUPLICATE_IMPORT_ALIAS` for any repeated alias in a module scope (two authored docs, owner + partial, or authored + inline), pinned to the second declaration ‚Äî replacing today's silent last-writer-wins. This newly rejects manifests that loaded before; accepted because silent alias shadowing is a latent bug and the project is pre-release. Not free, not folded into "reuse discovery verbatim."
- **Keep import execution in `import-controller.ts`.** The resolution path produces ordinary `Telo.Import` manifests, so the controller, multi-pass ordering, and teardown are reused verbatim ‚Äî the change is resolution-site only.
- **Editor: imports on the module-root right-side page; writes authored docs until the CLI follow-up.** Placement matches where module-level config now lives (#141) ‚Äî rejected a generic schema-driven node form (loses registry search and upgrade affordances) and keeping the UI in the sidebar (splits import management from the module config it belongs with). The write form stays authored `Telo.Import` docs and the docs keep recommending them, so the tool-written path stays CLI-manageable; both flip to inline in lockstep with the CLI follow-up. Rejected making inline the editor's write form immediately (the default path would then be the one the CLI can't upgrade) and pulling CLI into this PR (scoped as a follow-up to keep the core change focused).
