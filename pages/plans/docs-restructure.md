# Documentation Restructure

## Problem

The docs site reads as an unsorted bag rather than a learning path. The top level of [pages/sidebars.ts](../sidebars.ts) mixes a tutorial (Getting Started), how-tos (Guides), reference (Kernel, Std Lib, SDK), and orphaned single pages (Installation & CLI, Coding Agents) under no consistent organizing principle. "Standard Library" is a ~32-item near-alphabetical list with no semantic grouping — readers can't find a module by purpose unless they already know its name. Module depth is inconsistent: some entries are a single page, others are nested categories, with no visual cue. Kernel Reference is a 15-item flat list that defies scanning. Several public modules ([modules/codec/](../../modules/codec/), [modules/orm/](../../modules/orm/), the codec siblings) have no sidebar entry at all. Eleven runnable manifests in [examples/](../../examples/) are not surfaced in the sidebar at all. Module READMEs follow no shared shape, so each one feels like a different product. The net effect: new readers can't find a starting line, and returning readers can't predict where things live.

## Solution

Reorganize the sidebar into a Diátaxis-style five-group spine, surface the existing examples, and bring every module up to a shared README shape. Four layers of work, all in one pass:

**1. Sidebar IA.** Replace the current top-level list in [pages/sidebars.ts](../sidebars.ts) with five categories:

- **Learn** — Getting Started, Installation & CLI, Style Guide. (Today's "Guides" group dissolves into Learn entries; Testing and Coding Agents move to Build.)
- **Build** — task-oriented how-tos, populated from existing content: Testing your manifests (today's [tests/README](../../tests/README.md)), Working with coding agents ([docs/coding-agents.md](../../docs/coding-agents.md)), Deploying to AWS Lambda ([modules/lambda/docs/deploying.md](../../modules/lambda/docs/deploying.md)). Borderline pieces (Lambda cold starts, HTTP `returns & catches`, structured errors) stay in their module's Reference section and get linked from Build, not duplicated.
- **Reference** — split into two sub-categories:
  - **Kernel**: regrouped into _Concepts_ (Resource Definition, Lifecycle, References, Inheritance, Evaluation Context, Data Types), _Capabilities_, _Topology_, _Modules & Imports_ (Module System, Module Grants, Application Env Variables), _Runtime & Ops_ (Controllers, Signals, Telemetry & Observability).
  - **Standard Library**: regrouped by domain into _AI_, _HTTP & APIs_, _Storage & Data_, _Workflow & Control Flow_, _Encoding & Streams_, _Runtime Targets_, _Scripting_, _Configuration & Ops_, _Testing_. Within each group, modules sorted by typical reach-for order (most-used first), not alphabetically.
- **Extend** — SDK (Node.js, Rust). Future "Authoring a module" how-to slots in here.
- **Examples** — auto-generated index of [examples/](../../examples/) entries plus the nested [examples/aws/lambda/](../../examples/aws/lambda/) set. Each example surfaces its filename, the prose from its YAML `description:` field, and a link to the source. No per-example prose is written by hand.

**2. Examples index plugin.** A small Docusaurus plugin (sibling to the existing [pages/plugins/llms-txt/](../plugins/llms-txt/)) scans [examples/](../../examples/) at build time, parses each YAML's `metadata.description` field, and emits one markdown page that lists every example with its description and a GitHub source link. The plugin re-runs on rebuild so adding an example to [examples/](../../examples/) auto-surfaces it. [examples/aws/lambda/README.md](../../examples/aws/lambda/README.md) is wired in directly as the nested entry — no plugin handling needed.

**3. Missing module stubs.** Add minimal READMEs for [modules/codec/](../../modules/codec/), [modules/ndjson-codec/](../../modules/ndjson-codec/), [modules/octet-codec/](../../modules/octet-codec/), [modules/plain-text-codec/](../../modules/plain-text-codec/), [modules/sse-codec/](../../modules/sse-codec/), and [modules/orm/](../../modules/orm/), then wire them into the new Std Lib groups. Internal-only modules ([modules/dev/](../../modules/dev/), [modules/http-dispatch/](../../modules/http-dispatch/)) stay unindexed.

**4. Module README standardization.** Every module README is rewritten to a fixed five-section shape: title + one-sentence purpose; "Why use this" (3–5 bullets); kinds exported (one-line table); minimal usage example (single manifest); links to per-kind docs. Existing content is preserved where it already fits the shape; sections are reordered, trimmed, or added where they don't.

**URL changes.** Slugs move under the new groups: `/learn/<page>`, `/build/<page>`, `/reference/kernel/<page>`, `/reference/std/<module>/<page>`, `/extend/sdk/<lang>`, `/examples` and `/examples/<name>`. The current `/standard-library/...` slug injection in [pages/docusaurus.config.ts](../docusaurus.config.ts) `parseFrontMatter` hook is replaced with a `/reference/std/...` injector that mirrors the new sidebar tree. No redirects are added; this is a one-shot break called out in the changeset. The `llms-txt` plugin re-reads sidebars, so its bundles update automatically.

Per CLAUDE.md, no Docusaurus-only annotations (`sidebar_label`, `sidebar_position`, etc.) move into READMEs. All ordering and labels stay in [pages/sidebars.ts](../sidebars.ts).

## Decisions

- **Five top-level groups (Learn / Build / Reference / Extend / Examples).** Rationale: matches the Diátaxis split readers expect from modern docs sites (Next.js, Stripe, Tailwind), plus a dedicated Examples group because examples are complete artifacts, not patterns — folding them under Build or Reference understates them. Rejected three groups because we already have enough how-tos and runnable examples buried in the repo to populate Build and Examples on day one without writing new prose.
- **Build group populated only from existing how-tos; borderline pages linked, not duplicated.** Rationale: three clean how-tos (Testing, Coding Agents, Lambda Deploying) earn a real Build group; pulling explanation-flavoured pages (cold starts, structured errors, returns & catches) into Build would dilute it and create duplicate sidebar entries. Borderline pieces stay in module Reference where their context lives.
- **Examples surfaced via auto-generated index, not hand-written wrappers.** Rationale: each example's YAML `description:` field already explains itself; auto-generation means new examples appear without doc edits, and the burden shifts to writing good descriptions in the YAML (where the example author already is). Rejected per-example MDX pages because they'd need maintenance every time an example changes.
- **Standard Library grouped by domain, not alphabetically.** Rationale: readers searching by purpose ("how do I do HTTP?") outnumber readers searching by name. Within-group order is "most reached-for first" so the overview page of each group sits at the top.
- **Module READMEs standardized to a fixed five-section shape.** Rationale: predictability across modules is more valuable than per-module ergonomics; once readers learn the shape, every new module is instantly navigable. Rejected an opt-in template because inconsistent adoption is the current state and the symptom we're fixing.
- **Codec siblings and `orm` get stub READMEs and sidebar entries now.** Rationale: a module that exists but isn't documented is a worse signal than one with a thin "Overview + kinds" page.
- **`dev` and `http-dispatch` stay unindexed.** Rationale: they read as internal tooling; surfacing them in the public sidebar would confuse readers about what's stable. Re-evaluate if either becomes user-facing.
- **Breaking slug changes accepted, no redirects.** Rationale: the new URL scheme reflects the new IA and is the right long-term shape; maintaining a redirect map for a pre-1.0 docs site adds cost without protecting much external link surface. Called out explicitly in the changeset.
- **Plan lives at [pages/plans/docs-restructure.md](docs-restructure.md).** Rationale: the changes are concentrated in [pages/sidebars.ts](../sidebars.ts) + [pages/docusaurus.config.ts](../docusaurus.config.ts) + a new examples-index plugin under [pages/plugins/](../plugins/); module-README rewrites are spread across many modules but driven by the docs-site shape, so the plan belongs with the site config that anchors them.
