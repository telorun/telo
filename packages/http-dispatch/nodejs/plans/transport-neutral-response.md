# Plan ‚Äî Transport-neutral response rendering

Goal: extract the `returns:` / `catches:` rendering pipeline (currently `dispatchReturns` / `dispatchCatches` in [`modules/http-server/nodejs/src/http-api-controller.ts:199-440`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L199-L440)) so it no longer depends on the underlying HTTP transport implementation. **"Transport-neutral" here means independent of any specific HTTP server / response mechanism** ‚Äî Fastify today, AWS Lambda response envelopes next ([../../../../modules/lambda/plans/lambda-function.md](../../../../modules/lambda/plans/lambda-function.md)), future fetch-API / native `http.Server` / Workers adapters as they land. The dispatcher remains HTTP-shaped (status codes, headers, `Accept` / `Content-Type` negotiation, buffered vs streamed bodies); non-HTTP protocols (gRPC, MQTT, WebSocket, message queues) have different response vocabularies and are out of scope for this primitive.

**Status: stub.** This plan needs to be fleshed out before the Lambda function adapter can reuse the existing outcome-rendering schema. Filed here so the dependency is visible and the architecture decision isn't made in passing inside a downstream plan.

## Why this is its own plan

`dispatchReturns` and `dispatchCatches` are sophisticated:

- CEL-driven `when:` matching, status branching, schema validation, per-MIME content negotiation, buffer/stream modes, encoder ref injection, header merging, error-path fall-through.
- Analyzer-validated via `x-telo-outcome-list`, `x-telo-catches-for` ([`analyzer/nodejs/src/validate-throws-coverage.ts`](../../../../analyzer/nodejs/src/validate-throws-coverage.ts)) ‚Äî the schema is *the* outcome contract across the codebase.

But they're tightly bound to `FastifyReply` ‚Äî `reply.code`, `reply.header`, `reply.send`, `reply.hijack`, `reply.raw`, `reply.getHeaders`, `pipeline(..., reply.raw)`. To share with Lambda (or any other HTTP-shaped transport that doesn't speak Fastify), we need to:

1. Define a transport-neutral response sink (interface) that captures everything Fastify is being asked to do here ‚Äî status, headers, body, hijack-to-stream ‚Äî without leaking HTTP framework specifics.
2. Refactor `dispatchReturns` / `dispatchCatches` to write through the sink.
3. Provide a Fastify-backed adapter (preserves http-server behaviour) and a Lambda-backed adapter (translates sink calls into AWS response payloads).

This is cross-cutting infrastructure for HTTP-shaped transports. Designing it well *once* unblocks every future HTTP transport adapter; designing it ad-hoc per transport fragments outcome semantics across the codebase, which CLAUDE.md explicitly forbids.

## What the plan needs to cover

When this gets fleshed out, sketch the following:

- **Package boundary.** New sibling package `@telorun/http-dispatch`, sourced at [`packages/http-dispatch/nodejs/`](../../../../packages/http-dispatch/nodejs/). Depends on `@telorun/sdk` for the small handful of types it borrows (`ResourceInstance`, `Invocable`, `KindRef<Invocable>`, `InvokeError`, `Ref`); both `@telorun/http-server` and `@telorun/lambda` depend on it. The dispatcher does **not** live inside `@telorun/sdk`: the SDK is the universal module-author surface (every controller ‚Äî `modules/sql`, `modules/ai`, `modules/assert`, `modules/console`, ‚Ä¶) depends on it, and the SDK currently has **zero** runtime dependencies ([`sdk/nodejs/package.json`](../../../../sdk/nodejs/package.json)). Folding an HTTP-shaped dispatcher and `@sinclair/typebox` into the SDK would (a) ship HTTP content-negotiation code in the install tree of every non-HTTP module, (b) cross the one-way door of the SDK's first runtime dependency for a use case unrelated to module authorship, and (c) confuse the polyglot story ‚Äî a Python or Go SDK should not grow a `dispatch` subpath for symmetry, because dispatch is a transport-adapter concern, not a module-author concern. The new package imports **zero** Fastify symbols. The Fastify-backed adapter (`fastifyReplySink(reply)`) lives in `@telorun/http-server` next to the controllers that already import Fastify; the Lambda-backed adapter lives in `@telorun/lambda`. Adapters depend on `@telorun/http-dispatch`; the dispatcher depends on neither adapter. Lambda doesn't pull Fastify / `@fastify/cors` / `@fastify/swagger` / `@scalar/fastify-api-reference` ([`modules/http-server/nodejs/package.json`](../../../../modules/http-server/nodejs/package.json)) into its install tree just to render a `returns:` outcome.

  Splitting `dispatch` / `sink` / `schema` / `validate` lets the http-server controller import only what it needs (`fastifyReplySink` calls `dispatch`; `create()` calls `validate`) and keeps the package's public surface easy to scan. `test-utils/sink-contract.ts` is a *separate subpath export* (`@telorun/http-dispatch/test-utils`), not bundled with the main entry ‚Äî both transport adapters import it and run their own factory through it (see [Test parity](#test-parity)). Placement under `packages/` matches the existing pattern for cross-module foundational libraries (`packages/ide-support/`); it is not a Telo module (no resource kind, no `telo.yaml`) so it does not belong under `modules/`.
- **Sink interface.** Concretely:

  ```ts
  interface ResponseSink {
    setStatus(code: number): void;
    setHeader(name: string, value: string): void;
    send(body?: unknown): Promise<void>;
    stream(
      iter: AsyncIterable<Uint8Array>,
      onError?: (err: unknown) => void | Promise<void>,
    ): Promise<void>;
  }
  ```

  Coverage of every observable behaviour the existing dispatcher relies on:

  - **Status + headers.** `setStatus` / `setHeader` accumulate; the sink flushes on `send` / `stream`. Content-Type is just `setHeader("Content-Type", mime)` ‚Äî no special parameter. Header readback (currently `reply.getHeaders()` at [`http-api-controller.ts:281`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L281), used to feed `reply.raw.writeHead()` during hijack) is an internal concern of the Fastify adapter, not part of the public interface ‚Äî the dispatcher never reads accumulated headers back.
  - **Empty body** (`reply.send()` with no args at [`:226`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L226), [`:353`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L353)) ‚Äî call `send()` with no `body`. Both adapters render status + headers and end the response.
  - **Buffered body.** Dispatcher hands the body to `send(body)` *as-is* (`unknown`); each sink serializes per the accumulated `Content-Type`. The Fastify-backed sink calls `reply.send(body)` so Fastify's per-status response-schema dispatch through **fast-json-stringify** (registered at [`http-api-controller.ts:443-449`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L443-L449)) still runs ‚Äî no perf regression on the HTTP path. The Lambda-backed sink picks the right encoding for the AWS envelope: `Buffer` / `Uint8Array` ‚Üí base64 with `isBase64Encoded: true`; `string` ‚Üí pass-through `body`; everything else ‚Üí `JSON.stringify`. The dispatcher never encodes, never special-cases content type ‚Äî serialization concerns live only in the sink. The per-content-type serializer registry alluded to elsewhere becomes a *sink-side* extension point, not a dispatcher one.
  - **406 / 500 fallback envelopes** (currently inline JSON at [`:235-244`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L235-L244) and [`:341-348`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L341-L348)) ‚Äî the dispatcher renders these by setting status + `Content-Type: application/json` and calling `send` with the `{ error: { code, message, ... } }` envelope object; the sink serializes per the rules above. The envelope *shape* is part of the shared contract (every transport returns the same JSON shape on these paths); the sink doesn't need to know about them.
  - **Streamed body.** `stream(iter, onError?)` owns hijack / framing / `pipeline`-style propagation. The dispatcher hands over the already-encoded `AsyncIterable<Uint8Array>` (encoder ref invocation stays in the dispatcher per [Encoder ref injection](#encoder-ref-injection) below) and a callback. `onError` fires at most once if the iterator or the transport sink rejects after headers have flushed ‚Äî the response is already committed; the dispatcher can only observe, not `catches:`. This is the transport-neutral home for the existing `StreamErrorHook` ([`:89-92`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L89-L92)) that http-server uses to emit `Http.Api.streamFailed`; Lambda will route it to CloudWatch or an operator event from its own adapter. The callback signature is `(err: unknown) => void | Promise<void>` ‚Äî the dispatcher closes over `entry.status` / `matchedMime` at the call site and builds a per-event callback with that context baked in, rather than the sink interface carrying `{status, mime}` parameters it doesn't otherwise care about.
  - **Content negotiation.** `Accept`-header matching stays in the dispatcher (it drives which `content[mime]` is chosen and feeds CEL `result.*` access). The sink is told the result via `setHeader("Content-Type", mime)`, not via a parameter.
  - **No `commit()`.** Both `send` and `stream` end the response; the dispatcher never needs an explicit commit phase, and removing it eliminates a "did I forget to call commit?" footgun.

  All four methods return `Promise<void>` where async (`setStatus` / `setHeader` are synchronous accumulators). Adapters reject from `send` / `stream` only on transport-fatal errors (e.g. client socket closed before flush in buffer mode); mid-stream failures go through `onError` so the dispatcher doesn't need to distinguish "didn't manage to commit" from "committed then failed".
- **Refactor of `http-api-controller.ts` and `http-server-controller.ts`.** Move `dispatchReturns` / `dispatchCatches` (and the helpers `parseAccept`, `matchAcceptForMime`, `negotiateContent`, `matchEntry`, `applyHeaders`, plus the runtime schema-shape validators `validateNoContentTypeHeader` and `validateStreamWhenDoesNotReferenceResult`) into [`packages/http-dispatch/nodejs/src/`](../../../../packages/http-dispatch/nodejs/src/). The third validator in [`http-api-controller.ts`](../../../../modules/http-server/nodejs/src/http-api-controller.ts) ‚Äî `validateContentEntryShape` ‚Äî is **deleted**; its rule moves into the YAML schema (see the **Schema-expressed structural rules** bullet below for why only one of the three migrates). The TypeBox `ReturnEntry` / `CatchEntry` / `ContentEntry` types travel with the dispatcher. Replace `FastifyReply` parameters with the `ResponseSink` interface; provide a `fastifyReplySink(reply, { onStreamError })` adapter that stays in `@telorun/http-server` and is used at the call site inside http-server's own controllers. Two internal call sites migrate together: the `Api` route dispatch inside [`http-api-controller.ts`](../../../../modules/http-server/nodejs/src/http-api-controller.ts) and the `Server.notFoundHandler` dispatch at [`http-server-controller.ts:200-256`](../../../../modules/http-server/nodejs/src/http-server-controller.ts#L200-L256) (same `FastifyReply` coupling, same `dispatchReturns` / `dispatchCatches` import ‚Äî both flip to the sink in one go). `@telorun/http-server` adds `@telorun/http-dispatch` as a new workspace dependency and swaps direct imports of the local helpers for `import { ... } from "@telorun/http-dispatch"`.
- **Schema-expressed structural rules.** One of the three current runtime validators moves into the YAML schema as a `oneOf`-on-`mode` discriminated union; the other two stay as runtime guards in the dispatcher. The choice of `oneOf` over `if/then/else` is deliberate: `oneOf` matches the discriminator pattern already used elsewhere in the codebase and is first-class in editor form rendering. `if/then/else` would have required extending the editor's form generator for a single consumer.

  - `validateContentEntryShape` (body/encoder mutual exclusion; `mode: stream` requires `encoder` everywhere; `mode: buffer` forbids `encoder`; stream-mode requires a non-empty `content:` map) ‚Üí **`oneOf` on `mode`**. `ReturnEntry` becomes a discriminated union with two branches keyed on `mode: { const: buffer }` vs `mode: { const: stream }`. The buffer branch allows `content[mime].body` and forbids `encoder` per entry; the stream branch requires `encoder` on every `content[mime]`, forbids `body`, and sets `content: { minProperties: 1 }`. **Backward-compatibility caveat:** the existing `ctx.validateSchema` call site at [kernel/nodejs/src/resource-context.ts:123-126](../../../../kernel/nodejs/src/resource-context.ts#L123-L126) instantiates AJV with only `removeAdditional: true` ‚Äî **no `useDefaults`** ‚Äî so a naive `mode: { enum: [buffer, stream], default: "buffer" }` would *not* fill the default before `oneOf` evaluation, and manifests that omit `mode:` would fail to match either branch. To preserve backward compatibility without touching the kernel's shared AJV config (which would change validation behaviour for every other resource kind), the schema makes `mode` optional and expresses the rule as `oneOf: [ { ... mode absent OR mode: buffer ... }, { mode: { const: stream }, ... } ]`. Concretely the buffer branch uses `anyOf: [ { not: { required: [mode] } }, { properties: { mode: { const: buffer } } } ]` so existing manifests (which omit `mode:`) match the buffer branch unchanged. Zero new analyzer code; AJV's standard `oneOf` reporting surfaces the diagnostic.
  - `validateStreamWhenDoesNotReferenceResult` (stream-mode `when:` can't reference `result`) ‚Üí **stays as a runtime validator** in `packages/http-dispatch/nodejs/src/validate.ts`. The "naively express it as per-branch `x-telo-context` on the `when` field" approach does not work as-is: `extractContextsFromSchema` at [analyzer/nodejs/src/analyzer.ts:97-103](../../../../analyzer/nodejs/src/analyzer.ts#L97-L103) walks every `oneOf` branch and *unions* the collected `x-telo-context` entries at the same JSONPath scope ‚Äî it does not resolve the `mode` discriminator and pick one branch. So a buffer-branch context typing `result` as a regular value and a stream-branch context typing it as `x-telo-stream: true` would collapse into a merged, conflicting context for every `when:` field regardless of the entry's actual `mode`. Resolving this would require a real discriminator-aware `oneOf` walk in `extractContextsFromSchema`, which is out of scope for this plan. Keeping `validateStreamWhenDoesNotReferenceResult` as a runtime check is cheap, already works, and leaves the discriminator extension as future work that benefits every kind, not just this one.

  `validateNoContentTypeHeader` stays as a runtime validator in [`packages/http-dispatch/nodejs/src/validate.ts`](../../../../packages/http-dispatch/nodejs/src/validate.ts). The rule is expressible as JSON Schema `propertyNames`, but the runtime check is cheap, has a clearer error message than AJV's generic `propertyNames` failure, and keeps the dispatcher's defense-in-depth posture for manifests that somehow bypass `telo check`.

  The one schema-encoded rule rides along with YAML duplication: Lambda's per-source `$defs/Returns` / `$defs/Catches` (and every future HTTP-shaped transport) inherits the same enforcement for free ‚Äî the rule is *in* the schema text the consuming module copies, not in an http-server-only TypeScript module.

  **Follow-up not in this plan:** a discriminator-aware variant of `extractContextsFromSchema` (selecting the branch whose `properties.<discriminator>.const` matches the document's actual value when collecting `x-telo-context`) would let us move `validateStreamWhenDoesNotReferenceResult` into the schema too. Kind-agnostic improvement; worth its own plan once a second kind wants per-branch context typing.
- **Lambda adapter.** A sink that accumulates status / headers / body and returns the AWS Lambda response envelope when `send` completes. For streaming, wires into `awslambda.streamifyResponse` under managed runtime or chunked POST under custom runtime ‚Äî same `stream(...)` call site, different sink implementations. Lives in `@telorun/lambda` per the [Lambda plan](../../../../modules/lambda/plans/lambda-function.md).
- **Test parity.** Three layers, designed so transports can't drift in silence:

  1. **http-server regression.** All existing tests under [`modules/http-server/nodejs/tests/`](../../../../modules/http-server/nodejs/tests/) and the YAML suite under [`modules/http-server/tests/`](../../../../modules/http-server/tests/) must pass unchanged after the refactor ‚Äî same observable HTTP behaviour. No new test files at this layer; the refactor is meant to be invisible.
  2. **Sink contract harness.** `packages/http-dispatch/nodejs/src/test-utils/sink-contract.ts` exports `runSinkContract(makeSink: () => ResponseSink)` ‚Äî a vitest-style suite that exercises every method through a known sequence: status-only / empty body; status + headers + buffered body (assertions are on the *parsed* JSON envelope and the Content-Type, not byte-exact ‚Äî Fastify's fast-json-stringify and the Lambda sink's `JSON.stringify` can legitimately differ on whitespace / key order); status + headers + `stream` with onError callback fired after headers flush (here the streamed bytes *are* asserted byte-exact, since the dispatcher hands the sink an already-encoded `AsyncIterable<Uint8Array>`); double-`send` rejection; `setStatus` after `send` rejection. Both adapters (Fastify in `@telorun/http-server`, Lambda in `@telorun/lambda`) import the harness from `@telorun/http-dispatch/test-utils` and feed it a per-transport sink factory. Drift between transports surfaces as a contract-test failure, not a transport-specific bug discovered downstream. The harness lives behind a *separate subpath export* (not bundled with the package's main entry) so consumers don't drag test-only code into production bundles.
  3. **Lambda adapter unit tests.** Per-event-source AWS envelope assertions in [`modules/lambda/nodejs/tests/`](../../../../modules/lambda/nodejs/tests/), tracked by the [Lambda plan](../../../../modules/lambda/plans/lambda-function.md). These are *transport-shape* tests (`{ statusCode, headers, body, isBase64Encoded }` for API Gateway; `{ batchItemFailures }` for SQS; ‚Ä¶) and complement, not replace, the contract harness.
- **Schema portability ‚Äî duplicate inline.** The `returns:` / `catches:` schema lives inline in [`modules/http-server/telo.yaml`](../../../../modules/http-server/telo.yaml) (already duplicated within http-server for `Server.notFoundHandler` and `Api.routes[]`) and is *duplicated verbatim* into each consuming module's manifest ‚Äî Lambda's per-source `EventSource` definitions inline their own `$defs/Returns` / `$defs/Catches` copies that mirror the http-server shape exactly. No cross-module schema sharing: the existing Telo annotations (`x-telo-ref`, `x-telo-schema-from`) type field *values*, not schema fragments embedded in another kind's `$defs`, and `Telo.Type` doesn't splice into `$defs` today either. No lint, no codegen, no shared module to publish ‚Äî drift is on the author of each consuming module to keep in sync against the http-server copy. Code sharing (the `dispatchReturns` / `dispatchCatches` runtime) is unaffected: that ships via the `@telorun/http-dispatch` package described above; only the YAML schema is duplicated.
- **Streaming semantics across transports.** http-server commits headers then writes through `pipeline(Readable.from(...), reply.raw)`. Lambda has two streaming paths (managed-runtime `awslambda.streamifyResponse`, custom-runtime chunked POST). The uniform `stream(AsyncIterable<Uint8Array>, onError?)` shape covers both ‚Äî neither path is exposed at the dispatcher's call site. Backpressure-aware iteration (the `for await` chain in the encoder ‚Üí sink path, with `.return()` propagation on client disconnect, currently relied on at [`http-api-controller.ts:282-291`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L282-L291)) is the sink's responsibility per transport, not the dispatcher's.
- **Encoder ref injection.** The encoder is resolved at Phase 5 (`x-telo-ref` injection ‚Äî see [`http-api-controller.ts:259-274`](../../../../modules/http-server/nodejs/src/http-api-controller.ts#L259-L274)) and stays *inside the dispatcher*: when a `mode: stream` entry matches, the dispatcher calls `encoder.invoke({ input: handler.result.output })` to produce the `AsyncIterable<Uint8Array>` that the sink consumes via `stream(...)`. The sink never sees the encoder, the `Invocable`, or any kernel/SDK type ‚Äî it only ever takes bytes. This keeps resource resolution out of every transport adapter; if a future encoder needs different kernel coupling (cancellation tokens, backpressure-aware iterators, etc.), the sink contract stays the same.
- **Polyglot scope.** `@telorun/http-dispatch` is the Node.js implementation of the dispatcher. A future Go / Python equivalent ships as its own sibling package in that language (e.g. `telorun-http-dispatch` on PyPI, a Go module under `packages/http-dispatch/go/`) re-implementing the dispatcher against the duplicated YAML schema ‚Äî the YAML schema (`status` / `when` / `mode` / `headers` / `content[mime].{body,schema,encoder,headers}` plus the `x-telo-outcome-list` / `x-telo-catches-for` annotations) is the cross-language contract, not this TS package. Crucially, the Telo SDK in each language stays free of HTTP-shaped concerns ‚Äî module authors who never serve HTTP never pull dispatch logic into their dependency tree.
- **Changeset.** `@telorun/http-dispatch` ‚Äî new package, initial publish (0.1.0): ships `ResponseSink`, `dispatchReturns`, `dispatchCatches`, the TypeBox return/catch types, `validateNoContentTypeHeader`, `validateStreamWhenDoesNotReferenceResult`, and the `./test-utils` contract harness; runtime deps `@telorun/sdk` and `@sinclair/typebox`. `@telorun/sdk` ‚Äî **no change**: the dispatcher is not added to the SDK; the SDK retains its zero-runtime-deps posture. `@telorun/http-server` ‚Äî patch bump: controller-internal refactor onto the sink via `@telorun/http-dispatch` (added as a new workspace dependency); deletes `validateContentEntryShape` in favour of the `oneOf`-on-`mode` YAML-schema rule in [`modules/http-server/telo.yaml`](../../../../modules/http-server/telo.yaml); `validateStreamWhenDoesNotReferenceResult` and `validateNoContentTypeHeader` move into `@telorun/http-dispatch` as runtime guards (no schema migration ‚Äî see *Schema-expressed structural rules*); no observable behaviour change for valid manifests. Analyzer ‚Äî **no change** in this plan; the discriminator-aware `extractContextsFromSchema` extension is deferred to a follow-up plan. `@telorun/lambda` consumes `@telorun/http-dispatch` as a new workspace dependency in its own plan's changeset once it lands.

## Out of plan

- Beyond v1 transports. Once `http-server` and `lambda` both consume the sink cleanly, gRPC / WebSocket / MQTT can add their own adapters in their own plans without changing this primitive.
- A generic outcome-list runtime helper for transports that don't have HTTP-shaped responses (e.g. SQS batch-item-failure, EventBridge passthrough). These probably skip the sink and just return the invocable's output directly ‚Äî see [the Lambda plan](../../../../modules/lambda/plans/lambda-function.md) for how it handles non-HTTP event sources.
