# Module transports — abstraction + HTTP and OCI

Follow-up to [federated-registries.md](federated-registries.md) (host-qualified refs + inline integrity). Today a module is fetched, cached, version-listed, and published by logic that branches on the ref's shape, scattered across four sites. This plan **encapsulates every ref-scheme-specific behaviour behind one `Transport` abstraction**, then ships two implementations — the existing **HTTP registry** and a new **OCI** transport — so a publisher can push to any OCI registry they already run (GHCR / ECR / Docker Hub / Harbor) with **zero new infrastructure**. A third transport (S3, GCS, …) later becomes "implement one interface and register it." Discovery across federated hosts is a separate follow-up, [federated-discovery.md](federated-discovery.md).

## Problem

Transport-specific behaviour is smeared across the codebase with no unifying seam:

- **Resolution** — `ManifestSource` (`HttpSource`, `RegistrySource`) in `analyzer` is the one existing abstraction (browser-safe interface: `supports` / `read` / `resolveRelative`).
- **Cache keying** — a hardcoded if/else on ref shape in the kernel `local-manifest-cache-source` (`cachePathForUrl`).
- **Version enumeration** — a hardcoded `GET /<ns>/<name>` in the CLI `upgrade`.
- **Publish** — a bespoke single PUT (`pushToTeloRegistry`) in the CLI `publish`; the `publishers/` registry is for *controllers* (npm / cargo), not module artifacts.

Extending each of these four independently to add OCI would fan the divergence out further, and every new transport (S3 next) would repeat it. A correctness trap already lurks: an `oci://…@ver` ref satisfies the *registry-ref* shape-test in both `isRegistryRef` and `cachePathForUrl`, so without a scheme-owning seam it is silently misrouted to the default registry / a garbage cache path.

## Solution

**A `Transport` owns everything ref-scheme-specific about a module's lifecycle.** Registering a transport is the *only* thing needed to add a backend; the loader, cache, `upgrade`, and `publish` never branch on scheme again — they ask the transport registry which transport owns a ref and delegate.

A `Transport` **composes** a resolution `ManifestSource` with Node-only management methods — it does not extend or replace `ManifestSource`. That distinction matters: `ManifestSource` is the *resolution* primitive ("given a URL, get me the manifest bytes": `supports` / `read` / `resolveRelative` / `expandGlob`), and it is also implemented by things that are **not** transports — the on-disk cache (`local-manifest-cache-source`), `LocalFileSource`, `memory://` — which have no versions and nothing to publish. A `Transport` is the narrower concept of a *published, versioned, pushable backend keyed by a scheme*, so it adds the lifecycle methods that would be meaningless on those non-transport sources. Composition (not extension) keeps the cache / local / memory sources plain `ManifestSource`s.

**Resolution** — the transport's `ManifestSource` (browser-safe, stays in `analyzer`), unchanged: `supports(ref)`, `read(ref) → { text, source }` (fetch the manifest bytes and verify them against the inline `#sha256-…` hash via the shared `verifyIntegrity`), `resolveRelative`, `expandGlob`.

**Management** — Node-only, used by the CLI and the kernel cache:

- `fetchArtifact(ref) → { manifest, files }` — retrieve the **full artifact**: the `telo.yaml` *and* the bundled payload (`files:` → the module's shipped code/assets), verified against the `filesIntegrity` Merkle chain. This is what `telo install` calls; it **subsumes the out-of-band bundle fetch in `extract.ts`** that today sits entirely outside the source chain.
- `listVersions(ref) → string[]` — for `telo upgrade`.
- `publish(destination, bundle)` — for `telo publish`.
- `cacheLocation(ref) → string[]` — deterministic cache-path segments; the cache source (itself a plain `ManifestSource`) *consumes* this instead of computing the `cachePathForUrl` scheme-branch itself.

`read` returns the manifest only (the resolution/analysis path); `fetchArtifact` returns manifest **plus** payload files. The two exist because they serve different callers — the loader/analyzer/editor only need `telo.yaml`, while install needs the whole bundle — and because the **artifact shape differs by transport**, which `fetchArtifact` hides: an HTTP module is **two artifacts** (`telo.yaml` + a separate `module.tar.gz` payload, two GETs), an OCI module is **one artifact** (a single tar blob carrying both). Without this, an OCI-hosted module could not deliver its `files` at all — the payload is inside the blob `read` already pulls, but the abstraction exposed no way to hand it over.

The management methods live outside `ManifestSource` for a hard reason, not just tidiness: `publish` and `fetchArtifact` are inherently Node (read/write the bundle on disk, untar, run OCI's credential/token dance), but `ManifestSource` must stay browser-safe for the analyzer / editor — which never need payload files. So the seam runs between the resolution `ManifestSource` and the `Transport` wrapper, **not** between whole transports: **every** `Transport` class lives in `kernel` (it composes an analyzer-side source and adds the Node management methods), while the browser-safe resolution `ManifestSource` that a browser-reachable transport wraps (HTTP/registry) stays in `analyzer`; a Node-only transport (OCI's token handshake + Docker credentials; later S3's SDK) has no analyzer-side source at all. `kernel` — which already owns the cache source and loader wiring and is re-imported by the CLI, so runtime resolution and the CLI's publish/upgrade/install share one home. A transport declares whether it is **browser-reachable**; the editor resolves a non-reachable transport (OCI) through the discovery hub instead ([federated-discovery.md](federated-discovery.md)), never directly.

**The two implementations shipped by this plan:**

- **`RegistryTransport` (HTTP, default)** — bare `ns/name@ver` and `https://…`, default `registry.telo.run`. `read` = today's `RegistrySource` / `HttpSource`; `fetchArtifact` = `read` + GET `module.tar.gz` and untar (folding in `extract.ts`); `listVersions` = `GET /<ns>/<name>`; `publish` = the existing PUT (of `telo.yaml` + `module.tar.gz`); `cacheLocation` = `<path>/<ver>`. Browser-reachable — its resolution `ManifestSource` is the browser-safe `RegistrySource`/`HttpSource` in `analyzer`, but the transport wrapper (carrying `fetchArtifact`/`publish`) lives in `kernel`. A refactor of existing code behind the interface, not new behaviour.
- **`OciTransport`** — `oci://host/repo@ver`, over a **hand-rolled minimal OCI distribution client** (pull-manifest / pull-blob / push-artifact + `WWW-Authenticate` token flow + Docker credential chain; no external OCI library — see Decisions). `read` pulls the manifest descriptor + single tar blob, extracts `telo.yaml`, and verifies it against the inline hash (the OCI content digest is corroborating, not the source of truth — one integrity model across transports); `fetchArtifact` extracts `telo.yaml` **and** the payload files from that same blob in one round-trip. `listVersions` = the OCI tags API (`GET /v2/<repo>/tags/list`). `publish` pushes the bundle as one blob under an OCI **artifact manifest** (empty config descriptor `application/vnd.oci.empty.v1+json`, one layer `application/vnd.telo.module.v1+tar`, `artifactType` set to that media type). `cacheLocation` = `__oci/<host>/<repo>/<ver>`. **Not** browser-reachable (a browser can't do the token handshake / tar extraction / CORS). Auth reuses the ambient Docker credential chain (`~/.docker/config.json` + credential helpers); `docker login ghcr.io` is the only setup.

Both `read` paths funnel manifest bytes through the shared `verifyIntegrity` check; the bundle payload rides the existing `filesIntegrity` Merkle chain, re-verified on unpack. OCI only changes *how* the manifest bytes are fetched, not *which* bytes are hashed.

**Identity is the ref, not the manifest metadata — for every transport.** Where a module lives, and how it is imported, is the ref; `metadata.namespace`/`name` is a **display label and the internal kind prefix only**, never a location. Every transport method (`read`, `fetchArtifact`, `listVersions`, `publish`, `cacheLocation`) keys off the ref, never `metadata.name`. So a module whose `metadata.name` is `s3` can be pushed to `oci://ghcr.io/aws/telo-s3` and imported as exactly that ref with no mismatch — OCI decouples location from identity by construction, and HTTP now matches instead of forcing the path to equal metadata. Consequences: the **OCI destination is a full repo ref** (host-only `oci://ghcr.io` defaults the repo to `<ns>/<name>`); the **HTTP destination is a registry base** whose path defaults to `<ns>/<name>` but is authoritative once published — a *soft* publish-time check only warns when `registry.telo.run`'s path ≠ `<ns>/<name>`. `metadata.version` stays **authoritative** (changie owns it; publish gates on its movement) and must equal the published version/tag. **`metadata.namespace` becomes optional** and is slated for removal in a follow-up — it no longer has a structural role.

**Publishing selects the transport by the destination scheme — a destination-first positional.** `telo publish <destination?> <paths…>` (e.g. `telo publish oci://ghcr.io/aws/telo-s3 .`); the transport registry picks the publisher from the destination's scheme — `https://` / bare-host → `RegistryTransport`, `oci://` → `OciTransport`, omitted → default `registry.telo.run`. A positional is a *destination* only when it carries a scheme or looks like a host (dot-segment, no leading `./` or `/`, not `*.yaml`), otherwise it is a manifest path; `telo publish .` (or any directory) resolves to `<dir>/telo.yaml`, so the bare `telo publish .` keeps working, as does multi-manifest publish. Same `oci://` discriminator as imports, so one scheme drives resolve *and* publish — the destination-first, scheme-as-transport pattern of `git push <remote>` / `docker` / `rclone`.

**Relative imports resolve transport-relative, and canonicalize against the publish destination.** A relative import (`source: ../lib`) is resolved by `resolveRelative` on the source of its *containing* module — HTTP against the URL, registry against the registry base, OCI against the repo ref — so `local` is simply the base source for on-disk roots, never a transport. At publish, `canonicalizeImports` resolves each relative against the **destination** using the destination transport's `resolveRelative` and rewrites it to that absolute ref, so the pushed artifact carries no relatives (runtime never resolves `../lib` inside an OCI blob — it was rewritten away at publish, on the local tree). For OCI the repo ref is normalized to a **directory base** first, so `../lib` under `oci://ghcr.io/aws/my-app` resolves to `oci://ghcr.io/aws/lib` — not `oci://ghcr.io/lib`. This needs no metadata: the sibling's location comes from the destination, which matches what relative imports are *for* — co-developed siblings published under the same base (a dependency that lives elsewhere uses an absolute ref, not `../`). Because publishing an app does not publish its siblings, publish **verifies each derived ref resolves and fails on a dangling one** — the sibling must already be published, or published in the same invocation.

## Decisions

- **One scheme-keyed `Transport` for every ref-specific behaviour** — manifest read, full-artifact fetch, cache path, version list, and publish sit behind one scheme-keyed transport; adding S3 / GCS later is implementing it and registering it, with no new branch in the loader / cache / upgrade / publish / install. Rejected: extending the existing per-transport sites independently (fans divergence out, repeated per backend).
- **The transport owns payload retrieval, not just the manifest** — `fetchArtifact` returns `telo.yaml` **and** the bundled `files` payload, so an OCI module (whose payload lives inside the same blob) can deliver its code, and the out-of-band bundle fetch in `extract.ts` — which today sits entirely outside the source chain — folds into the transport. The abstraction hides the artifact-shape difference (HTTP: two artifacts; OCI: one blob). Rejected: leaving payload retrieval in `extract.ts` (OCI can't deliver `files`; the split persists per-backend). `read` stays manifest-only so the browser-safe resolution/analysis path is unaffected.
- **`Transport` composes `ManifestSource`, it does not extend it** — `ManifestSource` is the resolution primitive and is also implemented by non-transports (cache, local, memory); putting `publish` / `listVersions` on it would pollute those. `Transport` *has-a* resolution `ManifestSource` + Node-only management methods. This also respects browser-safety: `publish` is inherently Node, but `ManifestSource` must stay browser-safe for the analyzer / editor. Rejected: `Transport extends ManifestSource` (forces meaningless management methods onto cache / local / memory sources, and can't live in the browser-safe `analyzer`).
- **`Transport.supports()` owns the scheme — no ref falls through** — the hardcoded `isRegistryRef` / `cachePathForUrl` shape-tests are replaced by transport dispatch; `RegistryTransport` is the fallback only for bare / `https` refs, so an `oci://` (or future `s3://`) ref can never be misrouted to the default registry or a garbage cache path. Fixes the latent bug.
- **Publish / upgrade dispatch by scheme through the registry** — `telo publish` and `telo upgrade` pick the transport from the destination / ref and call `publish` / `listVersions`; no command hardcodes a transport. Destination-first positional, `.`/directory resolves to `<dir>/telo.yaml`. Rejected: a `--oci` flag (redundant with a scheme in the destination); folding the destination into `--registry` (a positional reads better and keeps multi-manifest publish).
- **Identity is the ref; `metadata.namespace/name` is a label** — location is the ref for every transport; `metadata.version` stays authoritative; `metadata.namespace` becomes optional and is slated for removal. *This is an identity-model change owned by [federated-registries.md](federated-registries.md); this plan consumes it and every `Transport` method keys off the ref.*
- **Relative imports canonicalize against the destination via `resolveRelative`** — publish rewrites `../lib` to the destination transport's resolved ref (OCI normalizes the repo ref to a directory base first, so `../lib` under `oci://ghcr.io/aws/my-app` → `oci://ghcr.io/aws/lib`), so pushed artifacts carry only absolute refs and the model needs no location metadata. Publish verifies each derived ref resolves and fails on a dangling one. Rejected: deriving the sibling's ref from its `metadata.namespace/name` (broken now that identity is the ref — the sibling's location isn't in its metadata); a sibling "publish-home" field or a batch destination-map (both redundant once relatives resolve against the destination).
- **Telo's inline hash stays authoritative across transports** — `read` verifies the extracted `telo.yaml` bytes against the inline hash via the shared `verifyIntegrity` (the OCI tar's own digest is corroborating), and the payload rides the existing `filesIntegrity` Merkle chain. One verification model regardless of transport.
- **OCI publish writes a standard artifact manifest** — empty config descriptor, one `application/vnd.telo.module.v1+tar` layer, `artifactType` set to that media type. Rejected: a bespoke manifest shape (breaks generic OCI tooling and the referrers API).
- **Editor OCI resolution is deferred to the discovery hub** — a browser can't speak OCI, so `OciTransport` is not browser-reachable and the editor resolves `oci://` imports via the hub's cached-manifest endpoint ([federated-discovery.md](federated-discovery.md)); this plan ships with the editor treating `oci://` as opaque. Rejected: injecting Node Docker-credential auth into `analyzer` (breaks browser-safety); an in-browser OCI client (impossible).
- **The Node side lives in `kernel`** — the transport registry + Node transports (`OciTransport`, later S3) live in `kernel`, which already owns the cache source and loader wiring and is re-imported by the CLI, so both runtime resolution and the CLI's publish/upgrade/install reach them without a new package. (`LocalFileSource` is Node-side too, for fs access, but it is a plain `ManifestSource`, not a transport.) Rejected: a dedicated `@telorun/transports` package now (boundary overhead + a changeset for two implementations); extract one only when a third+ transport lands. The seam is source vs transport, not browser- vs Node-transport: the resolution `ManifestSource` classes (`HttpSource`/`RegistrySource`) and the `ManifestSource` interface stay browser-safe in `analyzer`, but **every** `Transport` — `RegistryTransport` included — lives in `kernel`, because `fetchArtifact`/`publish` are inherently Node (untar, disk, PUT); a `Transport` composes its analyzer-side source and adds the Node methods, so it cannot itself sit in browser-safe `analyzer`.
- **The OCI client is a hand-rolled minimal client, not a dependency** — the JS OCI-client ecosystem has no usable option: ORAS ships no JS binding, the one capable client (`cloudydeno/docker-registry-client`) is Deno-only (violates the Node-only rule) and 4★, and its Node port (`oci-registry-client`) is ~3 years stale with uncertain push support. Telo's surface is small and rigidly specified by the OCI distribution spec v2 — pull-manifest, pull-blob, push-blob + artifact-manifest, the `WWW-Authenticate` token dance, and reading `~/.docker/config.json` / spawning `docker-credential-*`. Rejected: adopting a library (none maintained + Node-compatible; the Deno one would need porting anyway); pulling in a heavier client (supply-chain risk for a few hundred lines of well-specified code).

## Complete example after the change

An app mixing transports — the default HTTP registry and a publisher's own GHCR — every remote import pinned, all installed origin-direct:

```yaml
kind: Telo.Application
metadata:
  name: my-app
  version: 0.1.0
imports:
  Console: std/console@0.9.0#sha256-Yr4l2p...            # RegistryTransport (HTTP), default registry
  S3: oci://ghcr.io/aws/telo-s3@1.2.0#sha256-9Qk1mZ...   # OciTransport, publisher's own GHCR
targets:
  - !ref Console.writeLine
```

`telo publish oci://ghcr.io/aws/telo-s3 .` — the destination's `oci://` scheme selects `OciTransport`, which builds the bundle, pushes it as an OCI artifact to GHCR with the ambient Docker credentials, and treats the repo path as the identity, independent of `metadata.name`. `telo install` resolves each import through its transport (`RegistryTransport` for `std/console`, `OciTransport` for the GHCR ref), verifying the extracted `telo.yaml` against the inline hash. A future `s3://…` import would be a third `Transport` — no change to the loader, cache, `upgrade`, or `publish`. Making OCI modules discoverable across hosts (search, editor resolution of `oci://`) is [federated-discovery.md](federated-discovery.md).
