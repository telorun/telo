# OCI dual-publish — `ghcr.io/telorun` naming convention

Follow-up to [module-transports.md](module-transports.md) (the `Transport` seam + `OciTransport`) and [federated-discovery.md](federated-discovery.md) (the hub). Those shipped the machinery to publish and resolve a module over OCI; this plan settles **where Telo's own modules live on GHCR**, and makes the release pipeline push every stdlib module to both the HTTP registry and OCI.

## Problem

`scripts/publish-packages.mjs` pushes each changed manifest to `registry.telo.run` only. Adding OCI needs a repo-path convention, and the obvious one — mirror the registry ref, `ghcr.io/telorun/std/<name>` — is wrong for three independent reasons:

- **`std` is a registry ownership concept that OCI already expresses.** `apps/registry/telo.yaml` has a `namespaces` table with `owner_id → users(id)` because the central registry needed an ownership boundary. On GHCR the first path segment *is* that boundary — GitHub enforces write permission on the owner segment and nothing else. `ghcr.io/telorun/std/...` encodes "owned by Telo" twice.
- **`metadata.namespace` is being deleted.** [module-transports.md](module-transports.md) makes identity the ref and marks namespace "optional and slated for removal". A permanent OCI path must not depend on it.
- **Namespaces break relative sibling imports.** 24 stdlib modules import siblings relatively (`cache-redis → ../cache`). `OciTransport.resolveRelative` resolves against the repo as a directory base, so a namespaced layout puts `modules/lambda` (namespace `aws`) at `.../aws/lambda`, where its `../http-dispatch` canonicalizes to a dangling `.../aws/http-dispatch` and publish hard-fails.

The path choice is also effectively irreversible: the OCI distribution spec has no rename or alias mechanism, GHCR's REST API has no rename endpoint, and a public package past 5,000 downloads permanently retires its name if the org is ever renamed.

## Solution

**Publish to `oci://ghcr.io/telorun/<name>@<version>`** — one segment, flat, namespace dropped. `std/console@0.10.0` also becomes `oci://ghcr.io/telorun/console@0.10.0`.

The `ghcr.io/telorun` org root is **reserved for Telo modules and nothing else** — that is the decision the flat layout rests on. Container images live on Docker Hub (`telorun/node`, `telorun/registry`, `telorun/hub`) and stay there, so the root is unclaimed and every module name (`cache`, `sql`, `run`, `test`, `s3`) is free. Since GHCR hosts any OCI artifact, not only images, this reservation covers future charts, WASM builds, and dev containers too: they do not go in this org. The convention generalizes for third parties as `ghcr.io/<org>/<name>` — the illustrative `oci://ghcr.io/aws/telo-s3` in [module-transports.md](module-transports.md) and `apps/hub/telo.yaml` should be aligned to it.

**A one-time backfill bootstraps the mirror.** `telo publish` hard-fails when a canonicalized relative sibling does not already resolve at the destination, and 22 of the 52 stdlib modules import a sibling relatively — so the release path alone cannot establish an OCI mirror from empty. It only pushes manifests whose version moved, meaning a module whose sibling is absent fails and is not retried until its own version moves again, leaving the mirror permanently partial. `scripts/publish-oci-backfill.mjs` pushes the whole tree at current versions in dependency order and must run once against the destination **before** `TELO_OCI_REGISTRY` is set. It stops at the first failure rather than continuing, since every later module may depend on the one that failed. Re-running is harmless — same bytes, same tag, same digest — so it doubles as the repair path.

**Publish gains a second destination pass.** `scripts/publish-packages.mjs` already walks changed manifests in topological order (a sibling must be pushed before its dependents, since `telo publish` canonicalizes relative imports against the destination and verifies each resolves). OCI adds a second pass over the same ordered list, with the destination derived from the module's **directory name**, not its metadata — `telo publish oci://ghcr.io/telorun/<dir> <manifest>`. A new `TELO_OCI_REGISTRY` env var names the destination base, mirroring the existing `TELO_REGISTRY` — but with **no default**: unset skips the OCI pass entirely, and the release workflow sets it explicitly. Its presence is therefore the gate, with no second flag. Auth is `docker login ghcr.io` against `GITHUB_TOKEN` in the release workflow.

`OciTransport.publish` currently defaults a host-only destination's repo to `<namespace>/<name>`, which under identity-is-the-ref is wrong in general and would yield `ghcr.io/std/console` here — a `std` org Telo does not own. Because the pipeline always passes a full repo destination, that default is never exercised; it is removed rather than fixed, so a host-only destination is a clear error instead of a silently wrong path.

**The two published artifacts are intentionally not byte-identical.** Publish rewrites relative imports to destination-absolute refs and pins each with a `#sha256-` of the dependency's published manifest, so the OCI copy of `cache-redis` carries `oci://ghcr.io/telorun/cache@x#sha256-B` where the registry copy carries `std/cache@x#sha256-A`. Each artifact's dependency graph is internally consistent and stays on its own transport — which is what makes the OCI tree resolvable end-to-end by the editor through the manifest cache.

**Telo seeds the hub with its OCI refs only.** The hub itself stays transport-agnostic — it indexes whatever is registered, across `registry`, `oci`, and `url` alike — and this decision is only about what Telo puts in. The hub is empty today, so there is no migration: `SEED_REFS` (already an env-driven JSON array, empty by default) is populated with the `oci://ghcr.io/telorun/<name>` refs and the registry refs are never registered. **For Telo's own modules** that means one ref each — no duplicate rows, no duplicate embeddings, no competing hits in `/search/*`, `/refs`, or the `search_resources` MCP tool. **It does not bind anyone else: `/register` is open and per-ref, so a third party who dual-publishes and registers both refs produces exactly those duplicates. That is accepted — see the decision below.**

This makes **GHCR packages' public visibility load-bearing**, not hygiene: the hub tracker shells out to `telo module versions|digest|manifest`, and anonymous pull is what lets it read them without credentials. Publish therefore sets `org.opencontainers.image.source` (GHCR does not auto-link a package to its repo even when the path matches) and flips visibility to public per package. GHCR has no prefix-scoped visibility control, so ~50 modules means ~50 separately-administered packages — this must be scripted from the first release, not clicked through.

## Decisions

- **Flat `<name>` at the org root, namespace dropped** — the org segment already carries ownership, `metadata.namespace` is slated for removal, and a uniform sibling depth is what makes the 24 relative imports resolve. Rejected: `ghcr.io/telorun/std/<name>` (double-encodes ownership, breaks `aws/lambda`'s siblings, pins a dying field); a `modules/` infix (collision insurance against non-module artifacts in the org — unnecessary once the org is reserved for modules by policy, and a segment every ref pays for forever).
- **The `ghcr.io/telorun` org is reserved for modules** — this is what makes the flat layout safe, and it is a standing commitment rather than an observation: images stay on Docker Hub, and no chart, WASM build, or dev container goes in this org either, since GHCR hosts any OCI artifact. The cost of being wrong is unbounded — OCI has no rename or alias at spec, registry, or GHCR-API level, and an org rename permanently retires the names of public packages past 5,000 downloads.
- **No hierarchical scoping (`sql/postgres`, `http/server`)** — a relative import string is resolved against *both* the local filesystem and the publish destination, so nesting on OCI while the source tree stays flat makes the two disagree: `modules/sql-postgres`'s `../sql` canonicalizes to `telorun/sql/sql`, `cache-redis`'s `../cache` to `telorun/cache/cache`. Publish verifies derived refs and hard-fails, so this breaks every module with a sibling import rather than degrading quietly. Making it work would mean mirroring the nesting in the source tree and renaming each parent module out of its own directory — a repo-wide refactor of 24 import edges for path aesthetics. Lexical clustering in the GHCR package list (`sql-postgres`, `sql-sqlite`, `sql-repository`) already provides the grouping. Note the blocker is purely the relative-import arithmetic: a path that differs from the module name is fine under identity-is-the-ref.
- **Bare semver tags, no `v` prefix, no `latest`** — a `v` prefix breaks strict semver tag parsing across the ecosystem, and Telo's import grammar already rejects `@latest`, so a mutable tag would be dead weight. Note `+` is illegal in OCI tags, so build metadata would need escaping if changie ever emits it.
- **Destination derives from the module directory, never metadata** — identity-is-the-ref applied to publish; the directory name is already the de-facto module name in the release script's dependency ordering. The metadata-derived host-only default in `OciTransport.publish` is removed for the same reason.
- **A full-tree backfill precedes enabling the OCI pass** — the version-gated release path cannot bootstrap a mirror when 22 modules hard-fail on an absent sibling and are not retried until their own version moves. The backfill stops at the first failure (dependents would cascade identical "sibling does not resolve" errors and bury the cause), unlike the release pass which collects failures so one module cannot abort a release. Rejected: a flag on `publish-packages.mjs` (it opens with `changeset publish`, which must not run in a backfill); relying on topological order alone (it only orders *within* one release's changed set).
- **Manifest hashing lives on `Transport`, not in the CLI** — each transport hashes exactly what its own `read()` verifies (registry/HTTP raw bytes, OCI the UTF-8 text extracted from the tar layer), so a pin written at publish matches at import. Rejected: a scheme chain in `fetchManifestHash` — that *was* the bug, since an unhandled scheme degrades silently to unpinned, and it would recur with the next transport. Required on the interface, so a new transport cannot omit it.
- **`TELO_OCI_REGISTRY` has no default; unset skips the OCI pass** — one variable names the destination and its presence gates the pass, so dual-publish is an explicit deployment decision rather than a constant compiled into the release script. This matters more for OCI than for HTTP: the registry PUT fails safe without a Telo token, but OCI auth comes from the ambient Docker credential chain, so a fork or a developer who has run `docker login ghcr.io` for unrelated work holds usable credentials. Rejected: defaulting it to `oci://ghcr.io/telorun` (makes the pass unconditional, so it cannot gate anything and a fork's release run attempts a real push).
- **Telo seeds OCI refs only; duplicate third-party refs are accepted, not deduped** — Telo registers one ref per stdlib module, so its own corpus carries no duplicates. This is a seeding policy, **not an invariant**: nothing in `/register` or the schema stops a third party from registering both refs of a dual-published module and getting two rows, two embeddings, and two competing hits. Bounded only by there being no crawler — it takes deliberate double-registration.

  Deduping was considered and **deliberately rejected as a security trade, not a cost one**. Collapsing several refs into one hit that offers a pick-list makes search a supply-chain surface: whatever joins a group is presented to the user as an equal alternative to a trusted module, so a grouping mistake escalates from cosmetic noise to a malicious import. Rejected mechanisms: content-hash grouping (also broken on the merits — canonicalized imports make the two artifacts byte-different and the Merkle pin propagates that to every dependent, so it would collapse only import-free leaf modules); a derived recursive content key over import-normalized manifests (sound and unspoofable, but still builds the pick-list surface); a publisher-declared mirror set verified by reciprocity (same surface, plus it puts location data in the artifact, which `identity is the ref` bans). Duplicate refs appearing as separate results is the safe failure mode — each hit stands on its own ref, and the user chooses a location deliberately rather than from a list the hub asserts are equivalent.
- **`std/<name>` stays published and supported** — dual publish is additive; existing pins keep resolving. Because Telo seeds only its OCI refs and a configured hub *replaces* registry servers as the completion source, editor and CLI autocomplete will offer the OCI form, making it the de-facto recommended ref for new manifests. This is a deliberate soft migration, not a deprecation — no registry ref is withdrawn.
- **GHCR package visibility is part of the publish path** — anonymous pull is what lets the hub tracker read modules without credentials, so public visibility plus `org.opencontainers.image.source` is scripted per package rather than administered by hand.

## Complete example after the change

A release publishes `modules/cache-redis` (version `0.4.0`, importing `../cache`) to both destinations. The registry artifact carries `Cache: std/cache@0.6.0#sha256-A`; the GHCR artifact carries `Cache: oci://ghcr.io/telorun/cache@0.6.0#sha256-B`. An author then writes either form, and both resolve:

```yaml
kind: Telo.Application
metadata:
  name: my-app
  version: 0.1.0
imports:
  Cache: oci://ghcr.io/telorun/cache-redis@0.4.0#sha256-B  # hub-completed
  Console: std/console@0.10.0#sha256-C                     # still supported
targets:
  - !ref Console.writeLine
```

Telo seeds only the `oci://` ref for each of its modules, so `search_resources` returns each stdlib kind exactly once — not because the hub filters by transport, but because there is one seeded ref per module. The editor resolves the whole OCI import tree from `manifests.telo.sh` without ever speaking the OCI protocol from the browser.
