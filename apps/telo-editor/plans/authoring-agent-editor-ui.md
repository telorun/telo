# AI Authoring-Agent — Editor UI & Wiring

## Problem

The `authoring-agent` app is built: `POST /chat` with `{ conversationId, message }` streams back an `Ai.AgentStream` — a Server-Sent-Events sequence of `AgentStreamPart` records (`text-delta`, `tool-call`, `tool-result`, `finish`, `error`) — while editing manifests on its `./workspace` directory through `write_file` / `edit_file` / `read_file` / `list_dir` / `run` tools, validating every write with `telo check`. It has no UI, and the editor cannot talk to it, show its replies, or reflect the files it changes.

This plan delivers the complete editor integration: launching the agent, a two-way workspace-sync contract that is correct under every mutation path, a streaming chat panel, and the mode discipline that keeps the two in sync. It works for the browser editor (FSA / localStorage) and the desktop (Tauri) editor, and against any runner (local Docker or cloud) — the editor stays a thin client and the analyzer stays Node-free.

## Solution

### Launch & lifecycle

The agent is run like any other bundle, through a new **`AgentAdapter`** seam (`src/agent/`) parallel to `RunAdapter`, driven by an **`AgentProvider` / `useAgent`** context parallel to `RunProvider` / `useRun`, mounted in `src/main.tsx` beside `RunProvider`. The agent program — the `authoring-agent` app plus its module dependencies pinned to published versions — is embedded in the editor as a prebuilt `RunBundle` static asset, so the editor ships independently of the monorepo. The first chat message launches that bundle on the user's **active runner** via the existing run machinery (lazily — opening the panel does not launch it); the runner advertises the agent's port URL through the same `RunStatus.running` endpoints the run panel already consumes. The agent is a `keepAlive` service (never terminal), so it is not evicted by the runner's session TTL, and each user's session launches its **own** agent instance holding a private copy of that user's whole workspace — instances are isolated, with no multi-tenant sharing. The OpenAI key is **operator-provided**: the runner injects `OPENAI_API_KEY` at launch, read **only** by the in-process `OpenaiModel` controller and **scrubbed from every subprocess environment** the agent spawns (see Security) — so no agent tool can read or exfiltrate it. The user never supplies or sees a key and the feature is publicly usable out of the box (the editor collects nothing, and there is **no authenticated user**).

Because the feature is anonymous and operator-funded, spend is bounded server-side by two guards in the chat handler, both over a **shared `CacheRedis.Store`** (shared across every instance — per-instance memory would reset on each launch). The global bound is a new reusable **`RateLimit.Budget`** kind (added to `modules/rate-limit`: a windowed **weighted** counter over a `Cache.Store` — `RateLimit.Guard` only counts requests +1, so it can't sum tokens). It debits in **two phases** so the counter can't be gamed. At turn start `chatHandler` **atomically reserves** a worst-case per-turn cost (the model's `maxSteps` × max output budget) against the rolling window; an over-ceiling reservation is refused with "at capacity", and the *atomic* reserve closes the burst race where concurrent first-turns each read an under-ceiling total and all proceed. On **every terminal path** (`finish`, `error`, `Stop`, timeout — the transitions driven by the new turn-lifecycle owner in the `chat` library, see Streaming & reflection, not the clean-completion-only `RecordStream.OnComplete` tap) it **settles** the reservation to the turn's actual `finish.usage`, refunding the unused remainder — or keeping the full reservation when a turn errors or is cancelled with no usage report. Charging the reservation up front means a spam-start-then-cancel pattern still pays, so the counter never under-counts. A **per-client-IP** `RateLimit.Guard` gates alongside the budget check, its key the IP from the fronting proxy's `X-Forwarded-For` (passed into the handler from the route's `request` headers). The global cap is the hard guarantee that bounds operator cost — reserve-on-start + settle-on-any-terminal makes it leak-free; the per-IP limit is coarse (a rotated IP dodges it) but only slows an individual abuser, whom the global cap still stops. The agent trusts `X-Forwarded-For` only from its ingress proxy and is never directly reachable, so a client can neither forge its IP nor bypass either guard. A denial returns `retryAfter` and starts no turn.

### Security — anonymous and internet-reachable

Because the feature is anonymous and public, the threat model is **code execution and secret exfiltration**, not just request/token volume. Four measures bound it:

- **No command surface — no `run`, and `telo check` is not a tool.** The anonymous deployment exposes only `write_file` / `edit_file` / `read_file` / `list_dir`. `telo check` runs **automatically after every `write_file` / `edit_file`** (baked into the write/edit flow, returning `checkExitCode` / `checkOutput`), never as an agent- or user-callable tool — so validation always happens and the agent has no way to run an arbitrary command.
- **Key kept in-process, unset from subprocesses.** `OPENAI_API_KEY` is read only by the in-process `OpenaiModel` controller. The only subprocess the agent spawns — the auto-`telo check` — is launched with the key **removed from its environment** via a **new generic `shell` capability**: an `env: { KEY: null }` entry that *unsets* an inherited variable on `Shell.LocalHost` / `Shell.Command` (today `env` can only add or override a var, never remove one). Since no exposed tool executes agent-authored content *in-process* and the check runs in a **keyless subprocess**, no anonymous prompt can reach the key — no external proxy needed.
- **Injection-free validation.** The auto-`telo check` passes the file path as an **argv argument** through a **new generic argv-exec form on `shell`** — an optional `args: string[]` executed directly, **not** through `sh -c`. Today `Shell.Command` accepts only a single shell-string `command`, so a path is unavoidably concatenated (`checkCommand + ' ' + path`) and an agent-chosen path like `x.yaml; …` is shell injection even with `run` gone; the argv form removes the shell from the exec path entirely.
- **Container is the confinement boundary — no fs-level security capability.** `fs`'s `cwd` is explicitly *not* a security boundary and the module stays that way (a per-kind confinement capability would reverse that documented stance repo-wide, for a boundary the runtime already provides). Instead the **runner container** is the sole blast-radius boundary: the agent process runs confined inside it, so `read_file` / `list_dir` cannot reach operator infra, and outbound **egress is locked to the OpenAI API and the registry**. Confinement is an operator/runtime property, not an `fs` resource guarantee.

### Workspace-sync contract

The agent gains three HTTP routes (`Http.Api`, in `apps/authoring-agent`). The existing `fs` tools are all single-file (whole-file read/write/edit, `DirectoryListing` returns `{name, path, type, size}` with **no hash**, remove), so two of the routes cannot be composed from them and require **new generic `fs` primitives** (added to `modules/fs`, reusable for backup / mirror / deploy, per the generic-primitive rule):

- `Fs.TreeSnapshot` (new invocable) — a recursive content-**hash** walk of a directory → `{ files: [{ path, hash }] }`. `DirectoryListing`'s `size` is not a reliable change detector (equal-size ≠ equal-content), so a content hash is required. Backs `GET /workspace` (the tree manifest).
- `Fs.TreeSync` (new invocable) — apply an **explicit** change set `{ write: [{ path, content }], delete: [path] }`: write each file (creating parents) and remove each deleted path. It does **not** implicitly delete files absent from the set, so one route serves both a full seed (all files, empty `delete`) and a partial delta (only what changed) without wiping untouched files. Variable-length write + delete is real orchestration, not a static `Run.Sequence` over single-file tools. Backs `POST /workspace`.
- `GET /workspace/file?path=` — one file's contents; the only thin route (`Fs.File` read).

Sync covers the **whole workspace** — every application and library under `rootDir` — so one turn can create and edit across multiple modules (e.g. an app plus the library it imports). The editor walks `rootDir` via the `WorkspaceAdapter`, excluding vendor / build / cache dirs (`node_modules`, `.telo`, `.git`, `dist`), to form the seeded set. **Before each turn** the editor makes the agent's copy match its own files by diffing the agent's **actual** tree (`GET /workspace`) against the editor's current file hashes and pushing exactly that difference: missing or changed files in `write:`, paths present on the agent but gone from the editor in `delete:`. Seed-fullness is therefore derived from the agent instance's real state, **never from conversation age** — a fresh or restarted agent reports an empty tree and receives a **full seed**, a warm agent receives only the user's between-turn delta (edits, deletions, renames), and untouched files are never disturbed. Diffing against the agent's own tree (not a client-remembered "first message" flag) keeps seeding idempotent and correct across reloads, new devices, and agent restarts.

### Streaming & reflection

Chat is transported as a **resumable stream** so a turn survives a page refresh. `POST /chat` does **not** stream its response — it persists the user message, starts the agent turn **detached** into an in-memory replay journal keyed by a returned `turnId` (a new reusable `RecordStream.Journal` primitive — a keyed, offset-addressable buffer of `AgentStreamPart` records with monotonic ids), and returns quickly. The client then consumes **`GET /chat/{turnId}/events`**, an `EventSource` SSE stream that **replays the journal from `Last-Event-ID`** and then tails live — the same reconnectable shape the editor already consumes for runs, so `src/agent/` reuses `sse-client.ts`'s `lastEventId` + `sessionStorage` checkpoint machinery directly (a GET stream — no `fetch` + `ReadableStream` POST reader needed).

The detached turn is owned by a **new turn-lifecycle resource** in the `chat` library — the single authority for a conversation's in-flight turn, and net-new: today's `chatHandler` is a plain `Run.Sequence` that streams inline and tracks nothing, so none of this is derivable from the current agent. Per `conversationId` it holds the active `turnId`, the turn state (`running` → one of `finished` / `errored` / `cancelled` / `timed-out`), the bound `RecordStream.Journal`, and the reserved budget cost. Everything downstream derives from this one piece of state: `POST /chat` starts the agent turn into the journal, records `running`, returns the `turnId`, and **rejects an overlapping turn with 409** while one is `running`; `GET /conversations/{id}` exposes it for resume and lock-recompute; and a **terminal transition** — driven by any of four edges (the turn's own completion, an uncaught `error`, a user `Stop`, or a watchdog timeout) — fires exactly once to run the budget **settle** and finalize the journal. This terminal hook is exactly what `RecordStream.OnComplete` cannot be: `OnComplete` fires only on clean completion, so it cannot cover `error` / `Stop` / timeout — the lifecycle owner's transition does. The editor reflects the workspace **eagerly and then authoritatively**:

- On each mutating `tool-result` (`write_file` / `edit_file`) it pulls that one file via `GET /workspace/file` and applies it — live updates mid-turn, carrying the record's `checkExitCode` / `checkOutput` into the panel.
- On `finish` it runs a full **tree-hash reconciliation**: diff `GET /workspace` against its recorded hashes — changed/new files are pulled and written, files absent from the agent tree are deleted. This guarantees the editor's copy matches the agent's exactly after every turn **without replaying each tool's effect** (`edit_file` is a substring op, not a full-content event, so its result can't be reconstructed from the stream), and it's the single convergence primitive across reload / reconnect. Every application is content-hash-keyed, so reconnection or replay never double-applies.

All writes/deletes go through the editor's existing `WorkspaceAdapter` (`workspaceAdapterRef` — Tauri disk / FSA / localStorage) — the durable home — then through `afterFileMutation` (full `reloadWorkspace` when structure changed, else a tree refresh), letting the debounced `analyzeWorkspace` effect re-run. This reuses the exact reflect path `handleSourceEdit` already drives.

### UI panel

A dockable **chat side panel** (`src/components/agent/`) — a third column in the `flex-1` row of `src/components/Editor.tsx`, toggled from `TopBar`, available across every view.

The panel is **composed from [Vercel AI Elements](https://ai-sdk.dev/elements)**, not hand-built — a shadcn-registry component set installed via the shadcn CLI into the existing `src/components/ui/*` (its shadcn alias), the same way the current primitives were added. It is Radix-based and uses `lucide-react`, so it satisfies the UI-primitive rule by construction. The elements used: `Conversation` (stick-to-bottom transcript), `Message` / `MessageContent`, `Response` (streaming markdown assistant text), `Tool` → `ToolHeader` / `ToolInput` / `ToolOutput` (the tool-call **timeline**, one entry per call with a status badge, collapsible args, and result/error), `Reasoning`, `Loader`, and `PromptInput` / `PromptInputSubmit` (composer with a send/**Stop** toggle).

These are presentational only — they impose no runtime, so `AgentProvider` keeps owning the SSE connection and transcript, and a thin mapping renders each `AgentStreamPart`: `text-delta` accumulates into a `Response`; a `tool-call` and its matching `tool-result` render as one `Tool` entry (state `input-available` → `output-available`; `error: true` → error state; `write_file` / `edit_file` show the `checkExitCode` / `checkOutput` verdict in `ToolOutput`); `error` drives the error surface; `finish` closes the turn and drops the `Loader`. The composition + mapping live in `src/components/agent/`; the only new footprint beyond current deps is the streaming-markdown renderer `Response` pulls in.

### Turn-scoped locking, resume, and errors

`EditorState` (`src/model.ts`) gains persisted panel open/width; transient chat and lock state live in `AgentProvider`. There is no persistent "AI mode" — the chat panel is simply open or closed. Instead, writes stay strictly serialized in time by a **turn-scoped lock**: manual mutation is disabled only while an agent turn is in flight (from `send` to `finish`), and the workspace is free to edit between turns. This keeps the zero-merge correctness of a single writer while shrinking the critical section from a whole session to one turn (seconds) — no CRDT, base snapshots, or conflict UI.

`send` is the lock-acquire boundary: **persist** any dirty Monaco buffers, **re-seed the delta** to the agent (so it starts from the user's latest), **then** start the turn and lock. `finish` is the release boundary: reconcile agent→editor, then unlock. During the lock the user can still navigate, read, scroll, and inspect diagnostics, and watches the agent's writes land live via the eager per-`tool-result` reflection — only mutation is blocked.

The lock is **derived from turn liveness, never a persisted hard lock**, so it is self-healing and can never strand the editor: every terminal path releases it — `finish`, a stream `error` part, a non-2xx HTTP / launch failure, a turn timeout, and a user **Stop** control on the "AI working…" state (which cancels the turn and reconciles whatever landed). Errors are never swallowed; all of these render in the panel alongside non-zero `telo check` results.

One conversation per **workspace**, keyed by a stable workspace id (its `rootDir`) — never per Application, so a single thread spans every app and library the user is building; the editor generates and persists the `conversationId` and keeps its **own** display transcript (persisted client-side per `conversationId`). The agent persists model context in SQLite under the same id, so a reload or reconnect restores both halves consistently: on load the panel restores its transcript from local storage (the pending user message included), calls **`GET /conversations/{id}`** for turn status, and — if a turn is active — re-opens **`GET /chat/{turnId}/events`** from its last checkpoint: the half-written assistant text and tool calls **replay from the in-memory journal**, then streaming continues, so the user sees their message still pending and the answer resume. It also recomputes the lock from that status and runs a reconciliation pass. A mid-turn network blip reconnects the same way (`Last-Event-ID`); any gap is absorbed by the end-of-turn reconciliation.

Because one workspace conversation can have several attached clients (tabs, devices), turn serialization is enforced **server-side**, not only by the client lock: the agent tracks the in-flight turn per conversation (the same state `GET /conversations/{id}` exposes for resume) and **rejects an overlapping `POST /chat` with 409**, so two AI turns can never write the shared workspace at once. Every attached client also receives the in-flight turn on its stream and locks, so the client-side lock is a UX layer over an authoritative server guarantee. A manual edit made in an *unattached* tab during a turn is treated as an external change and converges to the agent's result on the `finish` reconciliation (hash-idempotent) — the manual-edit lock is scoped to a single client instance, the AI-turn lock is global.

## Decisions

- **Launch on the active runner, not a configured URL or shared filesystem** — the only model that works browser-sandboxed and against a cloud runner, and it reuses the proven run machinery and advertised-port endpoints. (Rejected: shared on-disk workspace — desktop-only and can't reach a browser/cloud agent; configured static URL — doesn't scale past local dev.)
- **Authoritative tree-hash reconciliation, not stream-derived edits alone** — reconstructing file state from `tool-call` args is fragile: `edit_file` is a substring replacement, not a full-content event, so any drift from the seeded copy would go unnoticed. Eager per-`tool-result` pulls give live UX; the `finish`-time tree diff guarantees exact convergence. Both are content-hash-idempotent. (Rejected: stream-only reflection — not provably convergent.)
- **Workspace seeded separately from the agent bundle** — the `RunBundle` carries only the agent program (pinned published module versions, embedded as an editor asset); the user's files travel via `POST /workspace`. Clean separation, and the pushed set is the whole workspace (all applications and libraries), so the agent edits across modules freely.
- **Conversation per workspace, whole-workspace sync — not per Application** — one agent instance per workspace holds a private copy of every application and library, so a single conversation creates and edits multiple modules at once (an app together with the libraries it imports). Keying per Application would fragment the workspace across instances and forbid cross-module edits — a core requirement. (Rejected: per-Application conversation / scoped seed — blocks multi-app / multi-library authoring.)
- **Sync needs two new generic `fs` primitives, not just routes** — `Fs.TreeSnapshot` (content-hash walk) and `Fs.TreeSync` (bulk write + explicit deletes) back `GET`/`POST /workspace`; the single-file `fs` tools can't express a content-hash tree or a variable-length diff. They live in `modules/fs` (reusable), and the agent exposes them plus a thin `Fs.File` read as the authoritative, browser-compatible sync contract — no bespoke wire protocol.
- **Anonymous exposure hardening — no command surface, key unset from subprocesses, container-confined** — an internet-reachable agent carrying the operator's key must assume code-execution and exfiltration attempts, not just volume. `run` is dropped and `telo check` is an automatic post-edit step (not a tool), so there is no arbitrary-command surface; the key is used only in-process by `OpenaiModel` and removed from the only spawned subprocess via a new generic `shell` `env: { KEY: null }` unset; the auto-check runs through a new generic `shell` argv-exec form (path passed as `args`, no `sh -c`, no injection); and confinement is the **container's** job, not `fs`'s — `fs` keeps its "not a security boundary" stance while the container confines the agent and egress is locked to the OpenAI API + registry. (Rejected: an operator egress proxy holding the key — a structural guarantee, but unnecessary once nothing executes agent content in-process and the only subprocess is keyless; keeping `run` — arbitrary-command surface for no authoring benefit; a per-kind `fs` confinement capability — reverses the module's documented stance for a boundary the container already provides; shipping anonymous without hardening — leaks the operator key.)
- **Anonymous public feature — global budget circuit-breaker + per-IP limit, no user key** — the runner injects the operator's `OPENAI_API_KEY`; there is **no authenticated user**, so spend cannot be metered per user. A new reusable **`RateLimit.Budget`** kind (windowed weighted counter over a shared `CacheRedis.Store`) that **reserves a worst-case cost at turn start and settles to actual `finish.usage` on every terminal path** — the atomic reserve closes the concurrent-burst race and charge-on-start closes the start-then-cancel leak, so it is a hard bound, not a soft one; a **per-client-IP** `RateLimit.Guard` adds throttling. `Budget` is a generic stdlib kind, not inline `JS.Script`. (Rejected: per-conversation / per-session key — client-rotatable, bounds nothing; per-user quota — no identity to key on; check-at-start / add-on-clean-completion — races and undercounts on cancel; client-side rate limiting — trivially bypassed.)
- **Resumable stream: POST starts a detached turn, a GET SSE replays it — turns survive refresh** — `POST /chat` returns a `turnId` and runs the turn into an in-memory `RecordStream.Journal` (monotonic ids); the client consumes `GET /chat/{turnId}/events` with `Last-Event-ID`, so a reload re-attaches and the pending message + streaming answer resume. It's the run-adapter's `GET /sessions/:id/events` shape, so the editor reuses `sse-client.ts` directly. (Rejected: streaming on the `POST` body — a POST stream can't be re-attached after refresh, which also made the plan's own "resume from last event id" claim unachievable.)
- **Chat UI from Vercel AI Elements, not hand-built** — a shadcn-registry set installed via the shadcn CLI (Radix + lucide, matching `src/components/ui/*`), with a first-class tool-call timeline (`Tool` / `ToolHeader` / `ToolInput` / `ToolOutput`) covering the "all tool calls on the timeline" requirement. It is presentational only, so it composes with `AgentProvider` by mapping `AgentStreamPart` records — no imposed runtime. (Rejected: hand-rolling chat primitives — needless and against the "install via shadcn before rolling your own" rule; assistant-ui — excellent tool UI but brings its own runtime/message model, which would fight the already-designed `AgentProvider` + SSE state.)
- **`AgentProvider` mirrors `RunProvider`** — same lifecycle, attach/resume, and persistence the editor already proves for runs; only panel open/width is persisted on `EditorState`.
- **Editor keeps its own display transcript; SQLite is the model's context** — the two restore independently under one `conversationId`, so reload and reconnect are consistent.
- **Turn-scoped locking, not single-writer-per-mode** — disabling manual edits only for the duration of a turn (with per-send delta re-seed and reconcile-on-finish) keeps writes serialized and merge-free, while letting the user edit freely between turns — a far better rhythm than a modal AI session. The lock is derived from turn liveness and released on every terminal path (finish / error / timeout / Stop), so it can never strand the editor. (Rejected: single-writer-per-mode — a coarse session-long freeze; concurrent dual-write with 3-way / CRDT merge — neither side emits fine-grained ops (whole-file writes and substring edits, not operations), so it buys a conflict-resolution problem for no structural need.)

## Data flow after the change

A user opens their workspace (browser or desktop) and opens the chat panel — no key or setup needed. They type "add an HTTP health endpoint" and send. The editor persists any dirty buffers, launches `authoring-agent` on the active runner (first send only), seeds the whole workspace, starts the turn, and **locks manual editing**. The assistant's reply streams in while tool-call cards show each `write_file` / `edit_file` and its `telo check` result; as each write lands the changed manifest updates live in the source and graph views with fresh diagnostics — the user reads and navigates but cannot type. When the turn finishes, a tree reconciliation guarantees the editor matches the agent exactly and the workspace **unlocks** — the user makes their own tweaks, then sends the next message, which re-seeds just those tweaks before the agent runs again. If a turn hangs or errors, the "AI working…" state's **Stop** button cancels it, reconciles, and unlocks. Reloading the page mid-turn re-attaches via `GET /chat/{turnId}/events`: the pending message and streaming answer resume from the in-memory journal where they left off. Reloading between turns restores the transcript, recomputes the lock from turn status, and reconciles.

## Correctness & edge cases

- **Bash-written / deleted / renamed files** — caught by the `finish`-time tree-hash reconciliation, not just the stream.
- **Reconnect / reload mid-turn** — the turn runs detached into an in-memory `RecordStream.Journal`; on reload the client re-opens `GET /chat/{turnId}/events` from `Last-Event-ID`, replaying the pending message and half-written answer, then tails live. The journal is in-memory, so an *agent* restart (not a client refresh) loses the live turn — the persisted final message + end-of-turn reconciliation recover the end state.
- **Duplicate/replayed events** — content-hash-keyed application is idempotent.
- **Unavailable runner** — surfaced as a pre-launch setup gate, never a silent no-op.
- **Secret exfiltration / RCE** — no `run` and no `telo check` tool (an automatic post-edit step), no exposed tool executes agent content in-process, the key is unset from the only subprocess (the keyless auto-check), and the check path is passed as argv with no shell (no injection) — so an anonymous prompt can neither read the key nor run arbitrary code; `fs` reads stay inside the runner container, whose egress is locked to the OpenAI API + registry. Residual: any *future* tool that executes agent-authored content in-process would reintroduce key exposure, so such a tool must instead run as a keyless subprocess.
- **Spend / abuse limits** — a per-IP `RateLimit.Guard` throttles an individual client and `RateLimit.Budget` refuses new turns once the operator's rolling-window ceiling is hit; either denial is surfaced with `retryAfter` and takes no lock (no turn started).
- **Start-then-cancel / concurrent-burst abuse** — cost is **reserved atomically at turn start** and only refunded down to actual on settle, so a cancelled or errored turn still pays its reservation and simultaneous first-turns can't all slip under the ceiling.
- **Isolation** — each session's agent instance holds a private copy of its own workspace; no cross-instance file or state sharing. The spend counters are the only shared state (a dedicated `CacheRedis.Store`).
- **Concurrent AI turns (multi-tab / multi-device)** — the agent rejects an overlapping `POST /chat` per conversation (409), so two AI turns can never write the shared workspace at once; the client-side turn lock is a UX layer over that authoritative guarantee, and any attached client reflects the in-flight turn and locks. A manual edit in an *unattached* tab during a turn converges to the agent's result on the `finish` reconciliation (hash-idempotent). Manual-edit serialization is scoped to one client instance; AI-turn serialization is server-enforced and global.
- **User edits / deletions / renames between turns** — on `send` the editor diffs the agent's actual tree (`GET /workspace`) against its current files and pushes an explicit `write:` / `delete:` delta, so creations, edits, deletions, and renames all propagate before the turn starts; the agent never works from a stale base.
- **Fresh / restarted / new-device agent** — seed-fullness is derived from the agent's actual tree, not conversation age: a reload or new session that hits an empty agent tree re-seeds fully, so a fresh instance is never left without its baseline files.
- **Stuck / hung turn** — the lock is derived from turn liveness (not a persisted hard lock) and released on every terminal path — `finish`, `error`, HTTP/launch failure, timeout, and a user **Stop** — so it can never strand the editor.
- **Multiple applications / libraries** — the whole workspace is seeded and reconciled, so one turn can author an app and the libraries it imports together; no per-Application scoping to collide.
- **Large workspaces** — the initial seed is the whole workspace minus vendor / build / cache dirs (`node_modules`, `.telo`, `.git`, `dist`); later syncs push only the changed-file delta, keeping steady-state cost low.
- **Structural vs. in-file changes** — routed through `afterFileMutation`, which already distinguishes full reload from a tree refresh.

## Housekeeping

Add a changeset for every touched published `@telorun/*` package. The new `fs` primitives (`TreeSnapshot` / `TreeSync`) bump `fs`; the new generic `shell` capabilities (argv-exec `args` form + `env: { KEY: null }` unset) bump `shell`; the `RateLimit.Budget` kind bumps `rate-limit`; and the resumable-stream `RecordStream.Journal` bumps `record-stream` — each via a changie fragment (with module `docs` + tests). The `chat` library changes bump `authoring-agent`: the `/workspace` routes, the resumable `/chat` contract (POST-start + `GET /chat/{turnId}/events` + `GET /conversations/{id}`) with its new turn-lifecycle owner, the budget + per-IP guards over a shared `CacheRedis.Store`, **and the security hardening — remove the `run` tool (and its system-prompt references), run the auto-`telo check` via the argv-exec form, and unset `OPENAI_API_KEY` from the check subprocess env**. Operator infra provides the shared Redis, the container confinement, and the egress lockdown (OpenAI API + registry only). Matching module docs under `apps/authoring-agent`.
