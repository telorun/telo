# Application Ports

## Problem

A Telo runner or the editor needs to know which inbound ports an application listens on **before** it starts the process — to display them, publish them, or reserve them. Today nothing declares this: the port lives inside a binding resource (e.g. `http-server.port: ${{ variables.port }}`) and is configured out of band via an env var, so a human separately "knows" which env var is the listen port. That knowledge is duplicated and not statically readable, and deriving it (static topology analysis or runtime events) either needs per-kind knowledge or comes too late.

## Solution

Add a top-level `ports` field to `Telo.Application` (sibling of `variables` / `secrets`), mirroring the variables mechanism so the env var is the directly-readable configurable knob.

`ports` is a name-keyed map; each entry declares `env` (host env var supplying the value), `protocol` (`tcp | udp`, optional, defaults `tcp`), and optional `default`. The value type is implicitly a port integer (1–65535) — no `type:` needed. Application-only; `Telo.Library` does not get `ports`.

- **Schema** — declare `ports` alongside `variables`/`secrets` in the `Telo.Application` builtin schema ([analyzer/nodejs/src/builtins.ts](../../../analyzer/nodejs/src/builtins.ts)). Map keys are unique by construction.
- **Load-time resolution** — a ports resolver mirroring `resolveBlock`/`coerce` in [src/application-env.ts](../src/application-env.ts), fixed to integer coercion plus the 1–65535 range check, applying `default` when the env var is unset and treating an entry with no default as required. Failures aggregate into the existing `ERR_MANIFEST_VALIDATION_FAILED`. Wired in next to the variables/secrets resolution in [src/kernel.ts](../src/kernel.ts).
- **CEL scope** — resolved ports surface as a new `ports` activation namespace (`ports.<name>` → integer) in [src/module-context.ts](../src/module-context.ts), so the binding resource reads `${{ ports.http }}` and the declaration is the single source of truth. Imported child modules do not receive `ports` (Application-only).
- **Analyzer typing** — register `ports` as a typed kernel global so `${{ ports.http }}` type-checks as integer and `ports.typo` is a diagnostic: add it to `KERNEL_GLOBAL_NAMES` in [analyzer/nodejs/src/kernel-globals.ts](../../../analyzer/nodejs/src/kernel-globals.ts), build its typed schema in [analyzer/nodejs/src/cel-environment.ts](../../../analyzer/nodejs/src/cel-environment.ts), and extend the wrapper-key strip in [analyzer/nodejs/src/residual-schema.ts](../../../analyzer/nodejs/src/residual-schema.ts) so the residual type is integer.
- **Editor** — surface declared `ports` into `ParsedManifest` ([apps/telo-editor/src/model.ts](../../../apps/telo-editor/src/model.ts)) so the run path can present them before launch. The editor already has `PortMapping` / `RunRequest.ports` / `RunnerEndpoint`; the declared list feeds those rather than replacing them.

## Phase 2 — Typed values (static wiring safety)

Phase 1 types `ports.<name>` as a plain integer. Phase 2 brands transport at the **value** level via a nominal `x-telo-type` annotation — `x-telo-type: TcpPort` / `UdpPort` on any value (a `variable` or a `ports` entry) — which the analyzer maps to a nominal CEL type. Standard JSON Schema keywords (`type: integer`, `minimum`/`maximum`) still do the real validation; `x-telo-type` carries only the nominal identity. Consuming resource fields carry the same brand, so wiring a `UdpPort` into a field branded `TcpPort` is a static analyzer error (`port 'dns' is a UdpPort but http-server.port expects a TcpPort`). This is general value typing, not a port-specific construct: it also covers port values that are never exposed (e.g. a DB-client port held in a plain variable), and the same machinery extends to other brands (`x-telo-type: Url`, `Duration`, …). Runtime is unchanged — the value stays `integer`.

- **Analyzer-only types** — map recognized `x-telo-type` brands to nominal CEL types registered on the analyzer's *cloned* CEL registry in [analyzer/nodejs/src/cel-environment.ts](../../../analyzer/nodejs/src/cel-environment.ts). `Registry.clone()` deep-copies, so these never reach the shared `@telorun/templating` base env the kernel runtime uses; cel-js `registerType` is used in its field-less form (no runtime constructor). The value is a bare integer at runtime, so only the analyzer's type-checker needs the nominal type — unlike `Stream`, whose instances flow at runtime and therefore must live in the shared env. An unrecognized `x-telo-type` is flagged (or falls back to `dyn`) so typos don't silently pass.
- **Brand-aware comparison** — `jsonSchemaToCelType` returns the branded type when `x-telo-type` is present, and the field-compatibility check in [analyzer/nodejs/src/schema-compat.ts](../../../analyzer/nodejs/src/schema-compat.ts) compares an expression's inferred nominal type against the consuming field's brand. The field's runtime JSON schema stays `integer`, so runtime validation is unchanged.
- **`ports` entries derive their brand from `protocol`** — an entry's Phase 1 `protocol` selects the matching brand (`tcp` → `TcpPort`), so `ports.http` types as `TcpPort` with no separate annotation. Purely additive over Phase 1; no kernel change.

## Unused-declaration warning

A declared port that no resource binds is misleading — the runner would advertise a port the app never listens on. A generic analyzer pass ([analyzer/nodejs/src/validate-unused-declarations.ts](../../../analyzer/nodejs/src/validate-unused-declarations.ts)) warns (`UNUSED_DECLARATION`) about any declared `variables` / `secrets` / `ports` entry that no CEL expression references. It walks every CEL expression once (both `${{ … }}` and `!cel`, via `walkCelExpressions`), extracts member-access chains, and buckets `<ns>.<name>` references by namespace in a single table-driven loop — no per-namespace branches. Dynamic / whole-namespace access (`ports[expr]`, `keys(variables)`) suppresses that namespace to avoid false positives. Application-only: a `Telo.Library`'s `variables` / `secrets` are a public contract consumed by its controllers (invisible to CEL), so they are not flagged.

## Decisions

- **Top-level `ports` field, not a literal/derived value.** A runner must know ports without starting the app, so the contract has to be explicit and static; static topology derivation was rejected (needs per-kind knowledge), runtime events were rejected (too late).
- **`env` named directly on each entry, not a CEL expression.** The knob the runner sets must be first-class data; encoding it as `${{ variables.port }}` would force the runner to parse CEL to find the env var.
- **Name-keyed map, mirroring `variables`.** Aligns with the existing variables shape and guarantees name uniqueness by construction, over a list-with-`name` form.
- **Mirrors variables semantics** (env-bound, load-time resolved, required unless `default`, same coercion/validation aggregation) to reuse the proven resolution path rather than inventing a parallel one.
- **Implicit integer + 1–65535 range, optional `protocol` defaulting `tcp`.** A port is always an integer in a fixed range, so `type:` is boilerplate; protocol is the only real variation and `tcp` is the common case.
- **New `ports` CEL namespace rather than folding into `variables`.** Keeps the binding resource pointed at a single declared source while leaving variables free of port-specific semantics and keeping the namespace statically analyzable.
- **Application-only.** A library is not a lifecycle participant and does not bind ports, so exposing the field there would be meaningless.
- **Phase 2 port types are transport-only (`TcpPort` / `UdpPort`), not application-protocol.** HTTP, gRPC, Postgres, etc. all ride TCP; per-application-protocol port classes (`HttpPort`) would explode combinatorially and wrongly reject a valid generic TCP port from feeding http-server.
- **Phase 2 typing is analyzer-only, registered on the cloned CEL registry.** `Registry.clone()` deep-copies, so the kernel runtime env is untouched, and a port's runtime value is a plain integer — only the type-checker needs the nominal type. Branded-int subtyping was rejected: it would require forking `@marcbachmann/cel-js`'s unification logic, which the closed primitive type system doesn't support.
- **Phase 2 typing is a general analyzer-only `x-telo-type` brand, not `format` or a port-specific field.** `format` was rejected because `tcp-port` / `udp-port` validate identically (integer 1–65535) — the distinction is a nominal brand, not a lexical format, and it has meaning only in the analyzer, which is exactly what `x-telo-*` annotations are for. A port-specific `x-telo-port` was rejected for not generalizing; `x-telo-type` extends to other brands (`Url`, `Duration`, …) and also type-checks non-exposed port values. Resolved by the analyzer with no hardcoded kind knowledge, preserving the topology-driven constraint.
- **`ports` is kept distinct from typed variables.** A type expresses value *shape* (a TCP port number), not *role* — a listen port and an outbound DB-client target are both `tcp-port`, so only an explicit exposure declaration tells the runner what the app listens on. `ports` is also Application-only, whereas `variables` span Application + Library, so an exposure flag on a variable would be asymmetric.
- **Unused-declaration check is a warning, generic across all three namespaces, and Application-only.** Warning (not error) so it never blocks `check`/CI or interrupts declare-then-wire editing. One table-driven pass covers `variables` / `secrets` / `ports` rather than three branches. Restricted to Applications because a Library's `variables` / `secrets` are a controller-consumed public contract — flagging them (e.g. `std/s3`'s credentials) would be a false positive.
