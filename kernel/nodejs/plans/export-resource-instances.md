# Export Resource Instances from Libraries

## Problem

A `Telo.Library` can export only **kinds**. A consumer that wants a stateless, zero-config resource — `Console.WriteLine`, which carries no config — must still declare a named instance just to give a `!ref` something to point at. In `examples/hello-world.yaml` that's a three-line `SayHello` block whose sole purpose is to instantiate a kind that has no per-instance state. Every consumer re-instantiates the same configuration-free singleton. The natural model — the one every real module system uses (`export const logger`, `System.out`) — is for the library to ship the ready-made instance and the consumer to just use it.

## Solution

A library declares its singleton internally and **exports the instance**; consumers reference it across the import boundary with the same `!ref` tag used for local references — one reference model, not a parallel mechanism.

**Exported instances.** `Telo.Library` gains `exports.resources` (a list of instance names), a sibling to `exports.kinds`. The Library schema in [analyzer/nodejs/src/builtins.ts](../../../analyzer/nodejs/src/builtins.ts) gains the array. Kind export and instance export are **independent gates** — a library may export a kind, an instance, or both. A library enforces a singleton by *choosing* to export the instance and omit the kind (no consumer can then construct a second one); a library that wants both a ready-made default and consumer-constructed variants (e.g. a `defaultLogger` instance plus a constructable `Logger` kind) exports both. Resolution is never ambiguous: a `kind:` / `extends:` slot resolves to the kind, a `!ref` slot resolves to the instance, so even a same-named kind and instance never collide at a slot. Any residual collision is a readability concern, handled by the casing convention below; a diagnostic for a same-name kind+instance export is a possible later addition, not required for correctness.

**Unified reference model via `Self`.** Every `!ref` is `<Alias>.<name>`. The tag carries a single source string; the resolver splits it on the **first dot** — `!ref Console.writeLine` parses to `{alias: "Console", name: "writeLine"}`, and a dot-free `!ref writeLine` normalizes to `{alias: "Self", name: "writeLine"}`. Dot-splitting happens only here, inside the `!ref` tag — never on untagged strings. The split is safe because the invariant it relies on is **enforced, not assumed**: `metadata.name` must contain no dot (a hard diagnostic — see Naming below), so the first segment is always either a known import alias or `Self`; an unknown first segment is an `UNRESOLVED_REFERENCE` error, never a silent mis-resolve into the wrong context. `Self` is already an auto-registered alias for the local module ([analyzer.ts:629-645](../../../analyzer/nodejs/src/analyzer.ts#L629-L645), used today by `extends: Self.<Abstract>`), registered through the same `registerImport` path as real imports. One resolver handles both cases: `Self` resolves against the local `byName` map (today's behaviour, unchanged); any other alias resolves into that import's exported resources, gated by `exports.resources` exactly as kind resolution is gated by `exports.kinds`.

**Sentinel resolution.** [analyzer/nodejs/src/resolve-ref-sentinels.ts](../../../analyzer/nodejs/src/resolve-ref-sentinels.ts) performs the `Self.` normalization and the alias-qualified lookup, rewriting `!ref` to the `{kind, name}` shape (carrying the resolving module/alias) so kernel injection can find the instance in the correct context.

**Library self-reference (`Self`).** A library that exports an *instance* must declare it internally as `kind: Self.WriteLine`, which requires `Self` to resolve the library's own kinds at both layers. Two changes: (1) the analyzer registers `Self` **ungated** (not bound to `exports.kinds`) — a library may reference its own kinds regardless of what it exports, since `exports` gates *importers*, not internal use; (2) the kernel registers `Self` → the library's own module in the import's **child context** (it has none today; only the analyzer knew `Self`), so `kind: Self.X` resolves at runtime. This is the prerequisite that lets a library drop a kind from `exports.kinds` while still instantiating it internally — enforcing the singleton (no importer can construct a second `WriteLine`).

**Kernel — materialization & cross-context injection (the novel path).** Materialization is *not* new: the create sub-phase of `initializeResources` ([evaluation-context.ts:232-270](../src/evaluation-context.ts#L232-L270)) eagerly creates **every** resource — stateless Invocables (`writeLine`, `readLine`) included, not only Services/Providers — and promotes each into the context's `resourceInstances`; `invoke()` later only looks up the already-created instance. So an exported instance is fully live in the child's `resourceInstances` once the import's `init()` (which runs `child.initializeResources()`) completes. What *is* new is exposing it across the boundary: Phase-5 injection (`injectAtPath` in [kernel.ts](../src/kernel.ts)) resolves a ref via `getInstance(name)` against the referencing resource's **own** context only. [import-controller.ts](../src/controllers/module/import-controller.ts) evaluates `exports.resources` and publishes the gated subset of the child's `resourceInstances` under an alias-keyed lookup. The load-bearing change is the **injection lookup signature itself**: today the closure is `getInstance(name)` ([kernel.ts:858](../src/kernel.ts#L858)), name-only and bound to the referencing resource's own `resourceInstances` — no alias parameter, no path into a child store. So the rewritten ref must carry the alias (`{kind, name, alias}`), `injectAtPath` must pass it (`getInstance(ref.name, ref.alias)` at [kernel.ts:1060](../src/kernel.ts#L1060)/[:1084](../src/kernel.ts#L1084)), and the closure must route a non-`Self` alias to the named import's published instances. This is a parent reading into a child import's instance store — a deliberate, export-gated **inversion of normal encapsulation**, and the genuinely novel part of the feature (not a benign analogue of the outward child-sees-parent scope-chaining in `createScopeHandle`).

**Kernel — init ordering.** `Telo.Import` is excluded from the dependency graph (`DEPENDENCY_GRAPH_SKIP_KINDS`, [system-kinds.ts](../../../analyzer/nodejs/src/system-kinds.ts)) and edge-drawing silently drops any edge whose target isn't a node ([dependency-graph.ts:88-98](../../../analyzer/nodejs/src/dependency-graph.ts#L88-L98)) — so the topo sort cannot order a cross-module ref, and neither the import nor the child instance is a node to point at. Ordering therefore rides the **existing multi-pass create/init interleave** in `initializeResources` (the same mechanism that orders imports today), not a new graph edge. The one required change: a cross-module ref that can't yet resolve must *defer* its referencing resource to a later pass — injection signals not-ready (the referencing resource stays in `createdInstances` and retries) rather than silently leaving the ref unresolved — and the loop converges once the import's `init()` has published its exported instances, within the same `MAX_PASSES` bound as every other resource. Rejected: promoting `Telo.Import` to a first-class graph node so the edge can be drawn — a larger change to the topo sort for ordering the multi-pass loop already provides.

**Lifecycle (unchanged).** No new lifecycle *concept*: the exported instance is the child context's existing singleton (created once, cached — [evaluation-context.ts](../src/evaluation-context.ts)), and teardown rides the existing child-context teardown and parent→child cascade. The inert `lifecycle: shared | isolated` field is untouched. The new work above is resolution and ordering, not lifecycle.

**CEL value-flow.** Exported instances are reachable from the importer's CEL as well, via the existing `resources.<alias>` surface — not a second reference mechanism. The import already publishes `resources.<alias>` as `{variables, secrets}` (import-controller `snapshot()`); this feature extends that aggregate to include each exported instance's own `snapshot()` under its name, so a value-flow export reads as `${{ resources.<alias>.<name> }}` (a config Provider or `defaultLogger`'s snapshot, say), gated by the same `exports.resources` list. This is what makes the primitive general rather than console-specific: value-flow exports (a config Provider, `defaultLogger`) are read in `${{ }}` and passed into other resources, while capability singletons (Invocable/Service/Mount) are consumed via `!ref` — two surfaces, one gate. Timing rides the same multi-pass convergence as the ref path: `resources.<alias>` is populated when the import snapshots (after its `init()` has materialized and snapshotted the child instances), so a parent CEL read defers until then. Exported instance names must not collide with the reserved `variables` / `secrets` keys under the alias.

**Analyzer validation.** [validate-references.ts](../../../analyzer/nodejs/src/validate-references.ts) validates an alias-qualified ref against the `exports.resources` gate and the slot's `x-telo-ref` kind constraint, with no resource kind hardcoded. This is *validation only* and does not feed init ordering: per the section above, imports are graph-excluded, so a cross-module ref draws no ordering edge. Surfacing the cross-module link for the editor's topology view is a separate, display-only concern.

**Naming — one enforced rule, one convention.** *Enforced (new hard diagnostic):* `metadata.name` must contain no dot. This is the load-bearing invariant the first-dot ref split depends on, so it is validated rather than left to convention (`metadata.name` is `{type: string}` today). *Convention only:* instance/resource names are camelCase going forward (especially exported ones); kinds, abstracts, and import aliases stay PascalCase — self-documenting, but unenforced, with no mass rename of existing manifests and no casing lint (a possible later step, out of scope). Casing buys readability; the dot-ban buys correctness — only the latter is enforced.

**std/console.** [modules/console/telo.yaml](../../../modules/console/telo.yaml) declares `writeLine` (and `readLine`) singleton instances and lists them in `exports.resources`; the `WriteLine` / `ReadLine` kinds are dropped from `exports.kinds`. `StreamWait` keeps its kind export — it has real config. [examples/hello-world.yaml](../../../examples/hello-world.yaml) drops the `SayHello` declaration and targets `!ref Console.writeLine` directly.

**Docs & changeset.** [modules/console/docs](../../../modules/console/docs) and the `exports` section of [CLAUDE.md](../../../CLAUDE.md) are updated; a changeset covers the affected published packages (kernel, analyzer, sdk if ref types change, the console module).

## Decisions

- **Kind export and instance export are independent gates — not a kind-XOR-instance rule.** A library exports a kind, an instance, or both. Singleton-ness is an *emergent choice* (export the instance, omit the kind), not a baked-in law — so the common "ready-made default + constructable kind" pattern (`defaultLogger` + `Logger`) stays expressible. Rejected: a hard XOR — its collision-removal justification is already supplied by the casing convention and by slot-context resolution (`kind:` vs `!ref`), and it would foreclose that pattern for a constraint we don't need; a kernel-level console singleton (breaks per-context `stdout`, redirection, isolated apps); exporting a live instance into the parent scope without an explicit gate (breaks import encapsulation).
- **Unify references through `Self`; bare `!ref name` is sugar for `!ref Self.name`.** One resolver (first-dot split inside the tag; dot-free → `Self`), reusing the existing auto-registered `Self` alias, instead of a second cross-module mechanism. `!ref` stays the single *reference* mechanism; CEL `resources.<alias>.<name>` is the complementary *value-flow* surface (below), not a competing reference path. Rejected: a dedicated instance keyword like `this` — the implicit qualifier is a namespace lookup ("this module"), which `Self` already means, while the genuine "current instance" token is the existing lowercase `self` (a resource's own config), so `this` would be both semantically wrong and a third redundant keyword.
- **Exported instances are reachable on both surfaces — `!ref` and CEL — under one gate.** `!ref`/`x-telo-ref` injects capability singletons (Invocable/Service/Mount); `${{ resources.<alias>.<name> }}` reads value-flow exports (Providers, computed values) via the existing `resources.<alias>` aggregate. These are Telo's two pre-existing consumption surfaces, not two competing reference mechanisms, so honoring both keeps the primitive general (covers `logger`/config, not just console). Rejected: ref-only — leaves value-flow exports (the motivating `defaultLogger`/config case) unreadable in expressions, making a "general instance export" half a primitive.
- **`Self` resolves a library's own kinds ungated, and is registered in the kernel's import child context.** Declaring an exported instance (`kind: Self.WriteLine`) needs self-kind resolution at runtime (the kernel had no `Self`) and must not be blocked by `exports.kinds` (which gates importers, not internal use). Strictly more permissive, so it can't break existing manifests, and it's what lets console drop the `WriteLine`/`ReadLine` kind exports to enforce the singleton. Rejected: keeping the kinds exported (singleton unenforced — a consumer could still construct its own).
- **No new lifecycle concept — but new resolution and ordering work.** Lifecycle is unchanged: exported instances ride the existing module-scoped singleton (one per import, shared across refs) and the existing teardown cascade; the inert `lifecycle: shared | isolated` field stays untouched. The genuinely new work is *resolution* (widening the name-only injection lookup to thread an alias and reach into an import's published instances) and *ordering* (imports are graph-excluded, so a cross-module ref rides the multi-pass create/init interleave with defer-and-retry, not a topo edge) — both promoted to first-class items in the Solution rather than folded into "lifecycle".
- **camelCase instances, convention-only (option B).** Makes cross-import refs self-documenting (`Console.writeLine` reads as a value, `Console.StreamWait` as a kind) and matches the type/value split already half-present (PascalCase kinds/aliases). Rejected: all-PascalCase (a `Console.WriteLine` ref is ambiguous to readers); enforced mass-migration with an analyzer rule (churn now, breaks existing user manifests) — deferred as a future option.
- **Plan and core mechanism live in the kernel/analyzer.** The reference resolver, export gating, injection, and validation are kernel- and analyzer-level; the console change is the first consumer, not the home of the feature.
